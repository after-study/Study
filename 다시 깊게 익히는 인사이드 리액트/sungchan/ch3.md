## 3.6 바닐라 vs 웹 컴포넌트 비교

1. UI 컴포넌트를 추상화핳 수 있습니다.
2. 재사용 가능한 컴포넌트를 작성할 수 있습니다.
3. 인터랙티비티를 제공하려면 여전히 명령형 프로그래밍을 사용해야 합니다.

### Shadow DOM 사용
- 명령형 자바스크립트로 셰도우 루트를 작성하여 섀도우 DOM 사용
- 브라우저에서만 실행 가능한 API이기 때문에 서버 사이드에서 섀도우 DOM 사용해 HTML 표현 불가
- 크롬 90, 에지 91 버전 부터 서버에서 선언적으로 섀도우 DOM을 작성하는 기능을 제공. 동적인 기능(이벤트 핸들링, 상태 관리)을 추가하려면 자바스크립트 명령형 코드 작성 필요
- 선언형 방식을 웹컴포넌트를 더욱 쉽게 활용하려면 Lit, Hybrid.js와 같은 웹 컴포넌트 프레임워크를 사용하는 것이 유용. Stencil.js, FAST와 같은 대안도 있음.

| 비교 항목 | 바닐라 자바스크립트 | 웹 컴포넌트 API |
|---|---|---|
| 정의 및 생성 방식 | 함수나 클래스를 사용하여 DOM 요소를 직접 생성 및 조작 | 표준화된 웹 컴포넌트 API(Custom Elements, 섀도우 DOM 등)를 사용하여 컴포넌트 정의 |
| 재사용성 | 코드 재사용을 위해 별도의 함수나 클래스를 만들어야 함 | 컴포넌트가 독립적인 단위로 정의되어 다양한 곳에서 쉽게 재사용 가능 |
| 캡슐화 | 변수나 함수의 스코프를 통해 일부 캡슐화 가능하지만 제한적 | 섀도우 DOM을 통해 스타일과 마크업을 완벽하게 캡슐화하여 외부와의 충돌 방지 |
| 스타일링 | 전역 CSS 또는 클래스 기반 스타일링 사용 | 섀도우 DOM 내에서 스타일을 정의하여 외부 스타일과 독립적으로 관리 가능 |
| 브라우저 지원 | 모든 현대 브라우저에서 지원 | 대부분의 현대 브라우저에서 지원하지만 구형 브라우저에서는 폴리필이 필요할 수 있음 |
| 학습 곡선 | 자바스크립트 기본 지식만 필요하지만 큰 규모에서는 관리 복잡 | 웹 컴포넌트 표준을 이해해야 하므로 초기 학습 장벽이 약간 있음 |
| 생태계 및 도구 지원 | 기본적인 자바스크립트 기능만 사용, 추가적인 라이브러리가 필요할 수 있음 | 웹 컴포넌트와 호환되는 다양한 라이브러리와 도구가 존재하지만 생태계는 프레임워크에 비해 작음 |
| 통합 용이성 | 어떤 프레임워크와도 쉽게 통합 가능 | 프레임워크에 독립적이지만, 특정 프레임워크와의 통합 시 추가 설정이 필요할 수 있음 |
| 퍼포먼스 | DOM 조작을 직접 관리하므로 최적화가 필요함 | 섀도우 DOM과 같은 최적화 기능을 제공하여 성능 관리가 용이함 |
| 유지보수성 | 코드가 복잡해질수록 유지보수가 어려워질 수 있음 | 컴포넌트 단위로 분리되어 있어 유지보수와 확장이 용이함 |
| SEO 친화성 | 서버 렌더링 시 SEO 최적화가 가능 | 클라이언트 측 렌더링이 기본이므로 SEO 최적화 추가 작업이 필요할 수 있음 |
