## 10.1 리액트의 프롭스와 컴포넌트 패턴을 돌아와야 하는 이유
- props는 단순 전달 도구 X, 강력한 활용 가능성과 잠재력 있음.
- props가 불변성 가져야 컴포넌트 예측 가능성을 보장. 
- 불변성의 본질과 일반객체 속성과의 근본적인 차이점 명확히 구분해야함.

- 고급 컴포넌트 패턴들의 차이점과 장단점 이해 -> props를 단순 전달 수단이 아닌 컴포넌트의 공개 API 설계 도구로 바라보는 관점을 갖게 됨.
- 컴포넌트 설계 능력 -> 재사용성 극대화 -> 협업 품질 개선.
- 잘 설계된 컴포넌트는 그 자체로 '사용 설명서'가 됨.

10장은 컴포넌트를 사용하는 개발자 -> 컴포넌트를 설계하는 아키텍트 의 핵심 디딤돌.

## 10.2 리액트의 프롭스와 상태 개념
props는 부모 컴포넌트로 부터 자식 컴포넌트로 데이터 전달하는 단방향 흐름을 담당.
컴포넌트 트리 구조와 props 데이터로 애플리케이션의 흐름을 이해할 수 있음.

### 10.2.1 리액트의 프롭스와 상태
부모 컴포넌트로 받은 props를 직접 수정할 수 없지만 useState(), useReducer()를 통해 자신만의 상태를 만들고 관리할 수 있음.
이 것을 다시 자식 컴포넌트로 전달하면 해당 상태를 props로서 받아보게 됨.

### 10.2.2 프롭스의 불변성
불변성을 강조하는 이유
1. 데이터 흐름을 예측가능하게 - 데이터 변경은 항상 상태를 소유한 컴포넌트에서 시작되므로 추적이 용이해짐.
2. 컴포넌트 동작을 예측하기 쉽게 - 같은 props는 같은 결과 렌더링되야 하므로 내부에서 변경되지 않아야 함.
3. 성능 최적화에 유리 - React.memo()와 같이 컴포넌트 렌더링의 최적화를 위해 프롭스 변경을 추적할 때 얕은 비교를 수행하므로 올바른 props 변경을 추적할 수 있도록 props를 전달받은 컴포넌트 내부에서 변경되지 않아야 함.

#### 잘못된 방법: 내부에서 변경
```tsx
const GoldenRabbitWrongButton = ({info}: {info: { status: string }}) => {
	const handleClick = () => {
		// 1. 프롭스로 받은 객체의 값을 직접 변경하려 시도함
		info.status = ‘reveal’;
	}
}
```

#### 불변성 유지하는 방법: 스프레드 연산자 사용
```tsx
const updateStatus = () => {
	setGoldieInfo({
		…goldieInfo,
		status: “reveal”
	})
}
```

### 10.2.3 프롭스와 속성
JSX props,  HTML attribute 의 차이
props는 HTML 표준에 맞춰 이름과 타입이 미리 정의되어 있으며, 렌더링 과정에서 실제 DOM 요소의 속성으로 변환됨.

커스텀 컴포넌트에는 원하는 이름으로 프롭스를 자유롭게 전달할 수 있음. 값으로는 모든 자바스크립트 값 사용. 훨씬 유연한 데이터 전달 수단임.
HTML 속성은 정적인 값(value = “initialValue”)이며, DOM 프로퍼티는 자바스크립트로 접근 가능한 DOM 객체인 값(inputElement.value)임. 리액트 프롭스는 대부분 DOM 프로퍼티에 대응됨.
이처럼 리액트는 DOM을 직접 다루는 대신 프롭스를 통해 선언적으로 UI를 관리함.

“@types/react”에서 정의된 HTML 요소의 속성 타입을 확인 할 수 있음.
```tsx
interface ButtonHTMLAttributes<T> extends HTMLAttributes<T> {
	disabled?: boolean | undefined;
	form?: string | undefined;
	formAction?: 
	// .. 생략
}
```

각 속성의 이름은 리액트 컨벤션에 맞춰서 카멜 케이스로 작성되는 것을 확인할 수 있음.

자바스크립트 예약어와 충돌하는 경우 대체 이름을 사용.
```js
class // className
for // htmlFor
```

### 10.3.1 런타임 검증을 위한 prop-types 라이브러리

리액트 15.5 버전 이전까지는 리액트 패키지에 포함. 이후 별도의 props-types 패키지로 분리. 19버전에서 제거.

```jsx
import PropTypes from ‘prop-types’;

// propTypes 정적 프로퍼티에 타입 지정
GoldenRabbitDetails.propTypes = {
	name: PropTypes.string.isRequired,
	age: PropTypes.number.isRequired,
	isHidden: PropTypes.bool
}

// props 기본값 설정
GoldenRabbitDetails.defaultProps = {
	isHidden: false
}
```

prop-types는 런타임에 타입을 검증하고 개발자 도구 콘솔에 경고 메시지를 표시함.
타입스크립트와 같이 빌드 또는 IDE에서 표기되는 정적 분석과는 다름.

## 10.4 프롭스를 사용하는 컴포넌트 패턴
재사용성을 극대화하는 핵심 도구
UI 구조화, 로직 공유, 렌더링 제어 등 특정 문제들을 우아하게 해결하는 데 도움을 줌.

- 기본적인 컴포넌트 합성
- children 프롭스의 다양한 활용법
- 렌더링 로직을 동적으로 주입하는 렌더 프롭스
- UI 특정 영역을 유연하게 채울 수 있는 슬롯 패턴
- 관련된 컴포넌트들을 하나의 단위로 묶어 관리하는 컴파운드 컴포넌트 패턴

### 10.4.1 컴포넌트 합성과 상속, 그리고 고차 컴포넌트

컴포넌트 간의 코드 재사용을 위해 리액트는 상속보다 합성 모델을 강력히 권장함.

- 합성(Composition): 여러 개의 작은 컴포넌트를 조립하여 큰 컴포넌트 만드는 방식. 컴포넌트 간의 결합도 낮추고, 독립성과 재사용성을 높임.
- 상속(Inheritance): 부모 컴포넌트 기능을 물려받아 확장하는 방식. 부모 자식간 강한 결합으로 유연성을 해치고, 프롭스가 여러 계층을 거치며 복잡하게 얽히는 문제를 야기할 수 있음.

합성의 가장 기본적이고 흔한 형태는 포함`Containment` 패턴.
```tsx
// …
return (
	<div>
		{children}
	</div>
)
```

또 다른 강력한 패턴은 고차 컴포넌트. 컴포넌트를 인자로 받아 새로운 컴포넌트를 반환하는 함수.

```tsx
// ch10_11/src/composition/HOCExample.tsx
const withLoadingSpinner = <P extends object>(WrappedComponent: ComponentType<P>) => {
	const ComponentWithLoadingSpinner: React.FC<P & WithLoadingProps> = ({ isLoadin, …props }) => {
		if (isLoading) {
			return <div>Loading…</div>
		}
		
		return <WrappedComponent { …(props as P) } />
	}
	
	// React 개발자도구에서 컴포넌트 이름을 명확히 하기 위한 설정
	const wrappedComponentName = WrappedComponent.displayName || WrappedComponent.name || ‘Component’
	
	ComponentWithLoadingSpinner.displayName = `widthLoadingSpinner(${wrappedComponentName})`
	
	return ComponentWithLoadingSpinner;
}
```

### 10.4.2 중첩 컴포넌트와 children, 프롭스 드릴링
합성 방식: children props를 활용해 컴포넌트 주입
중첩 방식: 부모 컴포넌트 내부에 자식 컴포넌트 직접 렌더링

#### 중첩 패턴
중첩 컴포넌트의 이점은 JSX구조만 봐도 관계를 이해하기 쉽다는 것. HTML의 중첩구조와도 비슷해 직관적.

```tsx
const NestedComponent = () => {
	return (
		<>
			<div>중첩 컴포넌트</div>
			{/* NestedComponent가 렌더링될 때마다 ChildComponent도 함께 생성되고 렌더링됨 */}
			<ChildComponent />
		</>
	)
}

```

#### 합성 패턴
children 프롭스를 활용, 부모 컴포넌트가 '자리'만 마련해주고 그 안에 들어갈 내용은 외부에서 주입받음.

```tsx
// children prop을 받아 렌더링하는 컴포넌트 정의함.
function ChildrenComponent({children}) {
  return (
    <>
      <div>children을 통한 합성</div>
      {/* 외부에서 이미 생성된 children 엘리먼트를 전달받아 렌더링만 함 */}
      {children}
    </>
  )
}
```
children props를 받아 지정된 위치에 렌더링만 할 뿐, children의 내용이 무엇인지 모름.

#### 중첩 vs 합성
jsx 함수로 변환되는 결과를 보면 차이를 확인할 수 있음
```tsx
// 애플리케이션의 메인 컴포넌트 정의함. ChildrenComponent와 NestedComponent를 렌더링함.
function MyApp() {
  return (
    <div>
      {/* 합성: MyApp이 ChildComponent를 생성하여 children으로 전달 */}
      <ChildrenComponent>
       <ChildComponent key="I'm children"/>
      </ChildrenComponent>
      {/* 중첩: MyApp은 NestedComponent만 생성 */}
      <NestedComponent></NestedComponent> 
    </div>
  );
}

export default MyApp;
```


```jsx
import { jsx as _jsx, jsxs as _jsxs, Fragment as _Fragment } from "react/jsx-runtime";

// MyApp의 JSX를 바벨로 변환한 결과 (개념적)
_jsxs("div", {
  children: [
    // ➊ MyApp이 ChildComponent 엘리먼트를 생성하고, 그 결과를 children 프롭으로 전달
    _jsx(ChildrenComponent, {
      children: _jsx(ChildComponent, {})
    }),
    // ➋ MyApp은 NestedComponent 엘리먼트만 생성. ChildComponent 생성은 NestedComponent의 역할
    _jsx(NestedComponent, {})
  ]
});

// NestedComponent의 변환 결과 (개념적)
_jsxs("div", {
  children: [
    _jsx("h3", {}),
    // ➌ NestedComponent의 렌더링 로직 안에서 ChildComponent 엘리먼트가 생성됨
    _jsx(ChildComponent, {})
  ]
});
```

합성 방식은 ChilrenComponent 가 렌더링 되더라도 children 프롭스 자체가 변하지 않았다면, ChildComponent는 리렌더링되지 않습니다. 렌더링 주체는 MyApp입니다.

중첩 방식에서는 NestedComponent가 리렌더링 될 때마다 그 내부에서 ChildComponent 엘리먼트가 새롭게 생성됩니다. 렌더링 주체는 NestedComponent입니다.

렌더링의 주체 차이를 이해하면 불필요하게 여러 계층을 거쳐 전달하는 프롭스 드릴링 문제를 해결할 수 있습니다.

children합성은 단순한 UI 구조화를 넘어, 컴포넌트 간의 결합도를 낮추고 불필요한 리렌더링을 방지하며, 프롭스 드릴링을 해결하는 강력하고 우아한 설계 패턴.

### 10.4.3 렌더 프롭스 패턴
렌더링 로직을 함수 형태의 프롭스로 주입하여 컴포넌트 동작을 제어하는 디자인 패턴

1. 컴포넌트의 특정 상태나 동작을 캡슐화하여 관리
2. 부모 컴포넌트로부터 프롭스로 전달받은 함수로 상태나 동작을 실행(위임)

이러한 제어의 역전을 통해 동작과 뷰를 완벽히 분리하여 재사용성과 유연성을 극대화할 수 있음.

```tsx
const RabbitPositionTracker: React.FC<RabbitPositionTrackerProps> = ({ children }) => {
	
	const [position, setPosition] = useState({ x: 0, y: 0 })
	
	const handleMouseMove = (event: React.MouseEvent<HTMLDivElement) => {
		setPosition({
			x: event.clientX,
			y: event.clientY,
		})	
	}
	
	return (
		<div>
			{children(position)}
		</div>
	)
}
```

position을 children으로 받은 함수에 넘겨주어 렌더링에 대한 제어권을 부모에게 넘김.

```tsx
<RabbitPositionTracker>
	{(position) => (
		<div
	style={{
		position: ‘absolute’,
		left: position.x,
		top: position.y,
		transform: ‘translate(-50%, -50%)’,
		fontSize: ‘2em’,
	}}		
>
		</div>
	)}
</RabbitPositionTracker>
```
children 프롭스로 함수를 넘겨 위치가 변경된 div를 반환.

RabbitPositionTracker는 마우스 위치를 추적하는 재사용 가능한 로직이 되었음.

렌더 프롭스 패턴은 꼭 children 프롭스를 사용해야 하는 것은 아님. render 프롭스에 함수를 주고 컴포넌트 내부에서 props.render(fetchedData)를 호출하여 화면을 그릴 수 있음.
```tsx
<DataLoader render={
	data => <Table rows={data} />
} />
```

렌더 프롭스 패턴은 컴포넌트가 렌더링할 내용을 함수로 전달받기에 렌더링을 동적으로 처리할 수 있으며 고차 컴포넌트와 같이 기존 컴포넌트를 재사용할 수 있는 강력한 패턴.

### 10.4.4 슬롯 프롭스 패턴
children props는 하나의 ‘통로’만 제공.
슬롯 패턴은 컴포넌트 자신의 레이아웃 안에 여러 개의 명명된 빈자리인 슬롯을 정의하고, 각 슬롯에 채워질 내용을 프롭스를 통해 외부에서 주입받는 패턴.

```tsx
interface ConfigurableModalProps {
	isOpen: boolean;
	onClose: () => void;
	headerContent?: ReactNode;
	bodyContent: ReactNode;
	footerContent: ReactNode;
}

const ConfigurableModal: React.FC<ConfigurableModalProps> = ({
	isOpen,
	onClose,
	headerContent,
	bodyContent,
	footerContent,
}) => {
	if (!isOpen) {
		return null;
	}
	
	return (
		<div className=“modal-overlay” onClick={onClose}>
			<div className=“modal-content”>
				{headerContent && (
					<div>
						{headerContent}
					</div>
				)}
				
				<div>
					{bodyContent}
				</div>
				
				{footerContent && (
					<div>
						{footerContent}
					</div>
				)}
			</div>
		</div>
	)
}
```
위 컴포넌트를 실제 사용하는 부모 컴포넌트 예제 생략

슬롯 패턴의 가장 큰 장점은 사용법이 명확하고 직관적. 단점은 슬롯의 개수와 이름, 전체적인 레이아웃이 고정되어 있어 유연성이 떨어짐.

### 10.4.5 컴파운드 컴포넌트 패턴
HTML의 \<select\>와 \<option\> 태그는 함께 사용될 때 비로소 완전한 드롭다운 컴포넌트가 될 수 있음.
컴파운드 컴포넌트는 서로 강하게 연관된 여러 컴포넌트를 하나의 그룹으로 묶어 이들이 공유된 상태를 기반으로 함께 동작하도록 만드는 것.
핵심은 부모 컴포넌트가 React.Context를 통해 공유 상태와 로직을 관리하고 자식 컴포넌트들이 이 컨텍스트를 구독하여 마치 하나의 단위처럼 유기적으로 작동하게 하는 것임.

```tsx
// ch10_11/src/compound-component/Tabs.tsx

interface TabsContextProps {
	activeTab: string | number;
	setActiveTab: (id: string | number) => void;
}

const TabsContext = createContext<TabsContextProps | undefined>(undefined);

const useTabs = () => {
	const context = useContext(TabsContext)
	if (!context) {
		throw new Error(‘useTabs must be used within a Tabs component’)
	}
	return context;
}

const Tabs: TabsComponent = ({ children, defaultValue }) => {
	const [activeTab, setActiveTab] = useState<string | number>(defaultValue);
	const contextValue = useMemo(() => ({ activeTab, setActiveTab }), [activeTab])
	
	return (
		<TabsContext.Provider value={contextValue}>{children}</TabsContext.Provider>
	)
}
```
useTabs() 커스텀 훅으로 자식들이 컨텍스트에 쉽게 접근하도록 돕고, Tabs 외부에서 잘못 사용되는 것을 막음.
Tabs 루트 컴포넌트는 useState()로 실제 활성 탭 상태를 관리하는 유일한 데이터소스.


```tsx
// ch10_11/src/compound-component/Tabs.tsx

// 개별 탭 버튼 (상태 소비자)
interface TabProps extends ButtonHTMLAttributes<HTMLButtonElement> {
  id: string;
}
const Tab: FC<TabProps> = ({ children, id, ...props }) => {
  const { activeTab, setActiveTab } = useTabs();
  const isActive = activeTab === id;
  return (
    <button
      role="tab"
      aria-selected={isActive}
      onClick={() => setActiveTab(id)}
      {...props}
    >
      {children}
    </button>
  );
};

// 개별 탭 패널 (상태 소비자)
interface TabPanelProps extends HTMLAttributes<HTMLDivElement> {
  id: string;
}
const TabPanel: FC<TabPanelProps> = ({ children, id, ...props }) => {
  const { activeTab } = useTabs();
  const isActive = activeTab === id;
  return (
    <div role="tabpanel" hidden={!isActive} {...props}>
      {children}
    </div>
  );
};

// 단순 컨테이너 역할의 컴포넌트들
const TabList: FC<{ children: ReactNode }> = ({ children }) => <div role="tablist">{children}</div>;
const TabPanels: FC<{ children: ReactNode }> = ({ children }) => <div>{children}</div>;

// 4. 하위 컴포넌트들을 정적 속성으로 할당 (API 설계)
Tabs.TabList = TabList;
Tabs.Tab = Tab;
Tabs.TabPanels = TabPanels;
Tabs.TabPanel = TabPanel;

export default Tabs;
```

Tab, TabPanel은 useTabs() 훅을 호출하여 공유된 상태에 접근하고 자신들의 상태를 제어

Tabs 컴포넌트에 Tab, TabPanel 등을 정적 속성으로 할당. 이는 기능적으로 필수적이지 않지만 API 설계를 통해 유저에게 “Tabs.”를 입력하는 것만으로도 관련된 모든 구성 요소를 자동 완성으로 찾을 수 있는 편리성 제공.

```tsx
// ch10_11/src/compound-component/Tabs.tsx
const App = () => (
  <Tabs defaultValue="tab1">
    {/* 탭 버튼 목록 */}
    <Tabs.TabList aria-label="샘플 탭">
      <Tabs.Tab id="tab1">탭 1</Tabs.Tab>
      <Tabs.Tab id="tab2">탭 2</Tabs.Tab>
      <Tabs.Tab id="tab3">탭 3</Tabs.Tab>
    </Tabs.TabList>

    {/* 탭 패널 내용 */}
    <Tabs.TabPanels>
      <Tabs.TabPanel id="tab1">
        <p>탭 1의 내용입니다.</p>
      </Tabs.TabPanel>
      <Tabs.TabPanel id="tab2">
        <p>탭 2의 내용입니다. 다른 내용을 포함할 수 있습니다.</p>
      </Tabs.TabPanel>
      <Tabs.TabPanel id="tab3">
        <p>탭 3의 내용입니다. 이미지를 넣을 수도 있습니다.</p>
      </Tabs.TabPanel>
    </Tabs.TabPanels>
  </Tabs>
);
```

이 패턴의 가장 큰 장점은 레이아웃 구성의 유연성이 높다는 것.

반면, 패턴 자체의 구현 복잡도가 높고, 컨텍스트 설정 등 초기 코드가 많이 필요함.

결론적으로 컴파운드 컴포넌트는 Tabs, Accordion, Menu 등과 같이 여러 부분이 유기적으로 상호작용하는 복잡하고 재사용성이 높은 UI 위젯을 만들 때 사용하기 적합한 패턴임.

### 10.4.6 명시적 슬롯 패턴 - slot type RFC`Request For Comments`
이전까지 학습한 패턴은 강력하지만 저마다의 트레이드 오프가 있음.
- 슬롯 프롭스: API 명확하지만 레이아웃이 부모컴포넌트에 고정되어 유연성이 떨어짐.
- 컴파운드 컴포넌트: Tabs.Tab 처럼 API가 선언적이고 유연하지만, 어떤 자식들이 허용하는지에 대한 계약이 암묵적이어서 잘못 사용될 여지가 있음.

컴파운드의 선언적 API와 슬롯 프롭스의 명시작 계약이라는 장점만 합친다면?
리액트 커뮤니티에서 명시적 슬롯 개념에 대한 논의가 진행중. create-slots와 같은 라이브러리를 통해 미리 탐색 가능.

이를 사용한다면 모달 컴포넌트 설계 시 슬롯 타입을 명시적으로 지정해줌으로써 의도에 맞게 컴포넌트를 올바르게 사용함과 동시에 컴파운드의 유연함까지도 달성 가능.
핵심은 컴포넌트가 자신이 받을 수 있는 슬롯의 종류를 명시적으로 정의하고, 유저는 그에 맞는 슬롯 전용 컴포넌트를 사용하여 내용을 채워 넣는 것.

```tsx
// ch10_11/src/create-slots/Modal.tsx
import { createHost, createSlot } from 'create-slots';

// ➊ 슬롯의 '타입' 또는 '설계도'를 명시적으로 정의함
const ModalHeader = createSlot('div');
const ModalBody = createSlot('div');
const ModalFooter = createSlot('div');

interface ModalProps {
  children: ReactNode; // 슬롯 컴포넌트들을 children으로 받음
  isOpen: boolean;
  onClose: () => void;
}

// 모달 컴포넌트 정의
export const Modal = ({ children, isOpen, onClose }: ModalProps) => {
  if (!isOpen) return null;

  // ➋ createHost를 사용하여 children에서 각 슬롯을 찾아 렌더링
  return createHost(children, (Slots) => (
    <div className="modal-overlay" onClick={onClose}>
      <div className="modal-content" onClick={(e) => e.stopPropagation()}>
        {/* ➌ 정의된 슬롯이 존재할 경우, 해당 위치에 렌더링함 */}
        {Slots.get(ModalHeader) && (
          <div className="modal-header">{Slots.get(ModalHeader)}</div>
        )}
        <div className="modal-body">{Slots.get(ModalBody)}</div>
        {Slots.get(ModalFooter) && (
          <div className="modal-footer">{Slots.get(ModalFooter)}</div>
        )}
        
        <button onClick={onClose} className="close-button">&times;</button>
      </div>
    </div>
  ));
};

// ➍ API의 편의성을 위해 슬롯 컴포넌트를 Modal의 정적 속성으로 할당
Modal.Header = ModalHeader;
Modal.Body = ModalBody;
Modal.Footer = ModalFooter;
```

1. createSlot() 함수로 Modal이 인식할 수 있는 슬롯의 종류를 미리 정의
2. createHost() 함수는 children 프롭스를 훑어보면서 슬롯 컴포넌트를 찾아냄. 그리고 두번째 인자로 전달된 콜백 함수에 Slots라는 헬퍼 객체를 제공.
3. Slots.get(ModalHeader)와 같은 코드를 통해 원하는 레이아웃의 위치에 배치
4. 컴파운드 패턴과 같이 정적 속성 할당

```tsx
// ch10_11/src/create-slots/App.jsx
<Modal isOpen={isOpen} onClose={() => setIsOpen(false)}>
	{/* ➊ 각 슬롯에 해당하는 컴포넌트를 사용하여 내용을 명시적으로 채움 */}
	<Modal.Header>
	  <strong>슬롯 패턴 모달 예제</strong>
	</Modal.Header>

	<Modal.Body>
	  <p>이것은 `create-slots` 라이브러리를 사용한 모달의 본문입니다.</p>
	  <p>슬롯을 사용하면 컴포넌트의 특정 영역에 내용을 명시적으로 전달할 수 있습니다.</p>
	</Modal.Body>

	<Modal.Footer>
	  <button onClick={() => setIsOpen(false)}>취소</button>
	  <button onClick={() => { alert('확인됨!'); setIsOpen(false); }}>확인</button>
	</Modal.Footer>
  </Modal>
```
\<Modal.Header\> 와 같이 매우 직관적이고 선언적인 방식으로 모달의 각 부분을 구성. 프롭스로 넘김 필요가 없음.

명시적 슬롯 패턴은 컴포넌트의 사용법을 강제하여 실수를 방지하고, 유연한 구성을 가능하게 하며, 뛰어난 개발 경험을 제공함.
