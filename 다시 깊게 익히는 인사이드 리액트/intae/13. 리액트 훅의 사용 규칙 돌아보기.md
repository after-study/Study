# Chapter 13 — 리액트 훅 살펴보기 정리

## 도입

리액트는 16.8버전에서 훅을 도입하여 패러다임의 전환을 맞이했습니다. 이는 단순히 새로운 API가 추가된 것을 넘어, 리액트 개발의 주축이 클래스 컴포넌트에서 함수형 컴포넌트로 이동하는 거대한 흐름의 시작점이었습니다.

파이버 아키텍처가 성능과 렌더링의 동작 방식 원리를 바꾸었다면, 훅은 개발자가 상태와 생명주기 같은 기능에 직접 연결하여 코드를 작성하는 방식 자체를 바꿨습니다.

---

## 13.1 리액트 훅을 돌아봐야 하는 이유

조건문 안에서 훅을 작성하면 안 된다고 동료에게 코드 리뷰를 해줄 수는 있지만, 정작 "왜 그렇게 해야 하죠?"라는 질문에 대해서는 머뭇거릴 수밖에 없는 스스로에게 의아함을 느낀 적이 있는지 모르겠습니다.

이번 장에서 다루는 것:

- 훅이 왜 '훅'이라 불리는지 개념적 의미부터 파헤침
- 클래스 컴포넌트의 한계점과 훅이 이를 어떻게 해결했는지 비교
- 훅을 올바르고 안전하게 사용하기 위해 반드시 지켜야 할 핵심 규칙 학습

## 13.1.2 this 가리키기 문제 — 클래스 컴포넌트의 한계

```js
class ThisContextExample extends React.Component {
  constructor(props) {
    super(props);
    this.state = { count: 0, message: "" };
    // ① 생성자에서 bind로 this를 고정한 새 함수 생성
    this.handleIncrementBound = this.handleIncrementUnbound.bind(this);
  }

  // ② 일반 함수: this = undefined
  handleIncrementUnbound() {
    console.log("Unbound this:", this); // → undefined
    this.setState((prevState) => ({ count: prevState.count + 1 }));
    // ❌ TypeError: Cannot read properties of undefined
  }

  // ③ 화살표 함수: 선언된 위치의 this를 그대로 사용
  handleIncrementArrow = () => {
    console.log("Arrow function this:", this); // ✅ 정상 참조
    this.setState((prevState) => ({ count: prevState.count + 1 }));
  };

  render() {
    return (
      <div>
        <h2>{this.state.message}</h2>
        <p>{this.state.count}</p>
        {/* bind 방식 */}
        <button onClick={this.handleIncrementBound}>증가 (bind)</button>
        {/* 화살표 함수 방식 */}
        <button onClick={this.handleIncrementArrow}>증가 (arrow)</button>
      </div>
    );
  }
}
```

- `handleIncrementUnbound`: 이벤트 핸들러로 넘기는 순간 `this`를 잃어버려 **TypeError** 발생
- `handleIncrementBound`: 생성자에서 `.bind(this)`로 `this`를 고정 → **정상 동작**
- `handleIncrementArrow`: 화살표 함수는 자신만의 `this`를 갖지 않고 선언된 위치의 `this` 사용 → **정상 동작**
- 이처럼 `this`를 다루기 위한 패턴이 클래스 컴포넌트를 복잡하게 만드는 주요 원인

### 잠깐 — JavaScript의 `this`와 `bind` 복습

#### `this`란?

`this`는 **함수를 누가 호출했느냐**에 따라 값이 달라집니다.

```js
const obj = {
  name: "리액트",
  getName() {
    console.log(this.name); // obj가 호출 → this = obj
  },
};

obj.getName(); // ✅ '리액트'

const fn = obj.getName;
fn(); // ❌ undefined — 호출한 주체가 없어서 this = undefined (strict mode)
```

> 클래스 컴포넌트에서 `<button onClick={this.handleClick}>` 처럼 메서드를 **이벤트 핸들러로 넘기는 순간**, 함수만 떼어져 나가기 때문에 `this`를 잃어버립니다.

---

#### `bind`란?

`bind`는 **`this`를 특정 객체로 고정**시킨 새 함수를 반환합니다.

```js
function greet() {
  console.log(this.name);
}

const user = { name: "철수" };

const boundGreet = greet.bind(user); // this를 user로 고정
boundGreet(); // ✅ '철수' — 어디서 호출해도 this = user
```

그래서 클래스 컴포넌트 생성자에서 이렇게 씁니다:

```js
constructor(props) {
  super(props);
  // 호출 위치와 상관없이 this가 항상 이 컴포넌트 인스턴스를 가리키도록 고정
  this.handleClick = this.handleClick.bind(this);
}
```

---

#### 화살표 함수는 왜 `this` 문제가 없을까?

```js
// 일반 함수: 호출 방식에 따라 this가 바뀜
function normal() {
  console.log(this); // 호출한 주체에 따라 달라짐
}

// 화살표 함수: 선언된 위치(렉시컬 스코프)의 this를 그대로 캡처
const arrow = () => {
  console.log(this); // 항상 선언 당시의 this
};
```

> 화살표 함수는 자신만의 `this`를 갖지 않기 때문에, 클래스 필드에서 선언하면 항상 **해당 클래스 인스턴스**를 `this`로 사용합니다.

> 이처럼 `this` 문제를 해결하기 위한 **bind, 화살표 함수** 패턴이 클래스 컴포넌트를 보일러플레이트 코드로 가득 차게 만들었고, 이것이 훅이 등장한 주요 동기 중 하나입니다.

## 13.3 자식 컴포넌트로부터 데이터 받기 — HOC 패턴

클래스 시절에는 로직 재사용을 위해 **HOC(Higher-Order Component)** 를 활용했습니다.

```js
// withWindowSize HOC가 windowSize props를 전달
class WrappedComponent extends React.Component {
  render() {
    return <WrappedComponent windowSize={this.state} {...this.props} />;
  }
}
export default WindowSizeViewer = withWindowSize(SizeViewer);
```

- `SizeViewer`가 `withWindowSize` HOC를 통해 `windowSize` props를 받는 구조
- HOC가 `withWindowSize()` → `SizeViewer` 순서로 window 크기를 주입
- **단점**: 컴포넌트 중첩이 깊어지고 구조가 복잡해짐

### 13.3.1 커스텀 훅을 사용하여 리팩터링하기

HOC를 커스텀 훅으로 대체하면 훨씬 단순해집니다.

```js
// ① window의 크기를 관찰하고 state로 상태 저장
function useWindowSize() {
  const [size, setSize] = useState({
    width: window.innerWidth,
    height: window.innerHeight,
  });

  useEffect(() => {
    const handleResize = () => {
      setSize({ width: window.innerWidth, height: window.innerHeight });
    };
    // ② 윈도우 리사이즈 이벤트 리스너 등록
    window.addEventListener("resize", handleResize);
    // ③ 클린업(cleanup) 함수: UI(OS)에 등록한 effects 정리
    return () => window.removeEventListener("resize", handleResize);
  }, []);

  return size;
}
```

```js
function WindowSizeViewer() {
  const { width, height } = useWindowSize(); // ① 커스텀 훅 사용
  return (
    <div>
      {width}x{height}
    </div>
  );
}
```

HOC 대신 커스텀 훅을 사용하면 `useWindowSize()`만 호출하면 되므로, 컴포넌트 계층 구조가 단순해지고 로직 재사용이 쉬워집니다.

## 13.4 라이프사이클 훅 사용기

### 훅의 내부 시뮬레이션 — `useState` / `useRef` / `useEffect`

리액트가 내부적으로 훅을 어떻게 관리하는지 파이버 기반으로 설명합니다.

```js
// 리액트 내부 시뮬레이션
let workInProgressHook = null;     // 현재 처리 중인 훅
let currentlyRenderingFiber = null; // 현재 렌더링 중인 파이버

function mountState(initialValue) { ... }  // ① 훅 최초 등록
function updateState(initialValue) { ... } // ② 훅 업데이트

function mountRef(initialValue) { ... }    // useRef 최초
function updateRef(create, deps) { ... }   // useRef 업데이트

function mountEffect(create, deps) { ... } // useEffect 최초
function updateEffect(create, deps) { ... } // useEffect 업데이트
```

**훅의 연결 리스트 구조:**

```
MyComponent의 파이버
↓
hook1(useState) → hook2(useRef) → hook3(useEffect)
[memoizedState]   [memoizedState]   [memoizedState]
```

- `mountWorkInProgressHook()`이 호출되면 훅 객체를 생성하고 **연결 리스트**로 연결
- `isMount`가 `true`이면 `mountState()`를, `false`이면 `updateState()`를 호출
- 이 연결 리스트가 **훅의 호출 순서가 항상 동일해야 하는 이유**
- `updateWorkInProgressHook()`에서 `currentlyRenderingFiber.memoizedState`가 `null`이면 `workInProgressHook`이 hook을 가리키며, 그렇지 않으면 다음 훅을 가리킵니다.

## 13.4.1 훅은 최상위에서만 호출되어야 한다

---

### 리렌더링 시 훅이 동작하는 방식

버튼을 클릭해 상태를 업데이트하여 MyComponent가 다시 리렌더링되는 시점을 생각해봅시다.
업데이트 시 리액트의 목표는 **이전에 만들어둔 연결 리스트를 순서대로 순회하여 각 훅의 상태를 가져오는 것**입니다.

이때는 이미 연결 리스트가 존재하므로, `useState()`의 경우 `mountState()`가 아닌 `updateState()`를 호출합니다.
나머지 훅들도 마찬가지로 `mountRef()`, `mountEffect()`가 아닌 `updateRef()`, `updateEffect()`를 사용해 각 훅 노드를 참조합니다.

> **단일 연결 링크드 리스트란?**
>
> 각 노드가 **데이터**와 **다음 노드를 가리키는 포인터(next)** 만을 가지고, 한 방향으로만 연결된 자료구조입니다.
>
> ```
> [Hook1 | next] → [Hook2 | next] → [Hook3 | next] → null
> ```
>
> 리액트의 각 Fiber 노드는 `memoizedState` 필드에 이 리스트의 첫 번째 훅을 저장합니다. 렌더링할 때마다 리스트를 **앞에서부터 순서대로** 순회하면서, **N번째 호출 = N번째 노드**로 매칭합니다. 키나 이름 같은 식별자 없이 오직 **호출 순서**로만 상태를 찾기 때문에, 조건문이나 반복문 안에서 훅을 호출하면 순서가 달라져 엉뚱한 상태를 읽게 됩니다.

```js
// 업데이트: 기존 훅 리스트를 순서대로 순회하는 것이 핵심
function updateState(initialState) {
  const hook = updateWorkInProgressHook(); // ① 기존 훅 노드로 이동
  const dispatch = (action) => {
    /* ... */
  };
  return [hook.memoizedState, dispatch];
}

function updateRef(initialValue) {
  const hook = updateWorkInProgressHook(); // ② 기존 훅 노드로 이동
  return hook.memoizedState;
}

function updateEffect(create, deps) {
  updateWorkInProgressHook(); // ③ 기존 훅 노드로 이동
}
```

리렌더링 시 다시 렌더링 시뮬레이터 함수인 `renderWithHook()`가 호출되고
이제 `isMount`는 `false`가 됩니다. `workInProgressHook` 포인터는 다시 null에서부터 시작하고
MyComponent 함수가 다시 실행됩니다.

---

### 리렌더링 시 훅 순회 단계별 흐름

```
hook1(useState) ──────→ hook2(useRef) ──────→ hook3(useEffect)
```

**1. `useState(0)` 호출:**

- `isMount`가 false이므로 `updateState()`가 호출됩니다.
- `updateWorkInProgressHook()`은 `workInProgressHook`이 null이므로,
  파이버에 저장된 리스트의 시작점인 hook1을 가져옵니다.
- `workInProgressHook` 포인터는 이제 hook1을 가리킵니다. hook1에 저장된 상태를 반환합니다.

**2. `useRef(null)` 호출:**

- `updateRef()`가 호출됩니다.
- `updateWorkInProgressHook()`은 `workInProgressHook`(hook1)의 next 포인터를 따라 hook2로 이동합니다.
- `workInProgressHook` 포인터는 이제 hook2를 가리킵니다. hook2에 저장된 ref 객체를 반환합니다.

**3. `useEffect(...)` 호출:**

- `updateEffect()`가 호출됩니다.
- `updateWorkInProgressHook()`은 `workInProgressHook`(hook2)의 next 포인터를 따라 hook3으로 이동합니다.
- `workInProgressHook` 포인터는 이제 hook3을 가리킵니다.

---

### 조건문 안에 훅을 넣으면 왜 문제가 생기는가?

컴포넌트가 리렌더링할 때마다 링크드 리스트를 순서대로 순회하며
기존에 저장해둔 노드에서 각 상태와 상태 업데이트 함수를 참조하기 때문에,
**만약 조건문 안에 훅을 넣으면 이 순서가 깨지면서 문제가 발생합니다.**

렌더링마다 훅의 호출 순서가 달라지면, 리액트는 `workInProgressHook.next`로
다음 노드를 찾아갔을 때 **자신이 예상했던 종류의 훅(`useState`)이 아닌
다른 종류의 훅(`useEffect`)을 만나게 될 수도 있습니다.**

이는 즉시 **예측 불가능한 버그와 애플리케이션의 충돌**로 이어집니다.
이것이 바로 리액트가 훅은 최상위에서만 호출해야 한다는 규칙을 강제하는 이유입니다.
이 규칙을 지키는 것은 리액트가 훅의 상태를 안정적으로 관리하기 위한 **최소한의 약속**입니다.

---

### 잘못된 예 — 조건문 내부에서 훅 호출 (규칙 위반)

```js
// ❌ ErrorComponent
// 오류: ① 조건문 내부에서 useState 훅을 호출
// showCounter값에 따라 훅 호출 순서가 달라질 수 있음
// 리액트는 훅 호출 순서가 매 렌더링마다 동일해야 함
function ErrorComponent({ showCounter }) {
  if (showCounter) {
    const [count, setCount] = useState(0); // ❌ 조건문 안에서 훅 호출

    // 마찬가지로 조건문 내부의 useEffect도 규칙 위반
    useEffect(() => {
      document.title = `Count: ${count}`;
    }, [count]);

    const handleIncrement = () => {
      setCount((prevCount) => prevCount + 1);
    };

    // 조건부 렌더링 자체는 문제가 없으나, 훅 호출이 문제임
    return (
      <div>
        <p>현재 카운트: {count}</p>
        <button onClick={handleIncrement}>증가</button>
        <p>카운터가 표시됨 (훅 규칙 위반)</p>
      </div>
    );
  }

  // showCounter가 false이면, 위의 훅들은 호출되지 않음
  // 따라서 렌더링 간 훅 호출 순서가 달라짐

  // 예시: 조건문 밖에서 호출되는 다른 훅 (규칙 준수)
  const [anotherState, setAnotherState] = useState("기본값");

  return (
    <div>
      <p>카운터가 표시되지 않음.</p>
      <p>다른 상태: {anotherState}</p>
      <p>이 컴포넌트는 조건문 내부에 훅을 사용하여 규칙을 위반했음.</p>
    </div>
  );
}
```

> 훅이 잘못 작성된 예입니다. 조건문 안에서 훅이 선언된 경우 `showCounter` 변수에 의해
> **동일 컴포넌트의 서로 다른 스냅샷에서 훅이 선언되거나 선언되지 않는 경우가 발생합니다.**
> 이렇게 훅이 조건문 안에서 작성되면 안 됩니다.
>
> `ErrorComponent`와 같이 조건문으로 훅을 선언하면 스냅샷 1에서는 3번 인덱스가 `useRef()`까지만
> 스냅샷 2에서 `useEffect(0)`으로 변경될 경우 리액트가 상태를 올바르게 추적할 수 없게 됩니다.

### 올바른 예 — 훅 규칙을 준수한 CorrectComponent

```js
// ✅ CorrectComponent
function CorrectComponent() {
  // useState 훅을 컴포넌트 최상위 레벨에서 호출
  // ① 매 렌더링 시 같은 순서로 호출되는 것을 보장함
  const [count, setCount] = useState(0);

  // ② useEffect 훅 또한 조건문이 아닌 위치에서 호출함
  useEffect(() => {
    // 문서의 타이틀을 현재 count값으로 업데이트함
    document.title = `Count: ${count}`;
    // 이 효과는 count값이 변경될 때마다 실행됨
  }, [count]);

  const handleIncrement = () => {
    setCount((prevCount) => prevCount + 1);
  };

  return (
    <div>
      <p>현재 카운트: {count}</p>
      <button onClick={handleIncrement}>증가</button>
      <p>이 컴포넌트는 훅 규칙을 올바르게 준수하여 작성되었음.</p>
      <p>useState와 useEffect 훅은 항상 같은 순서로 호출됨.</p>
    </div>
  );
}
```

이 컴포넌트는 `showDetails` 같은 외부 요인과 관계없이, 렌더링이 몇 번을 반복되더라도
훅의 호출 순서는 항상 **① `useState()` → ② `useEffect()`** 로 동일하게 유지됩니다.

이처럼 예측 가능한 순서를 보장함으로써, 리액트가 내부적으로 사용하는
**연결 리스트 기반의 상태 관리 메커니즘이 안정적으로 동작**하도록 할 수 있습니다.

> 조건부 로직이 필요하다면, **훅의 호출 자체를 제어하는 것이 아니라
> 훅의 내부나 JSX에서 처리**해야 합니다.

## 13.4.2 오직 리액트 함수 내에서만 훅을 호출해야 한다

훅은 반드시 **리액트 함수 컴포넌트** 또는 **커스텀 훅** 안에서만 호출해야 합니다. </br>
일반 JavaScript 함수, 클래스 컴포넌트, 이벤트 핸들러 등에서는 호출할 수 없습니다. </br>
왜냐하면 훅이 동작하려면 현재 어떤 컴포넌트를 렌더링하고 있는지에 대한 정보인 렌더링 컨텍스트가 필요하기 때문입니다.
리액트는 이 컨텍스트를 통해서 훅의 상태를 올바른 컴포넌트와 연결합니다.

```js
// ❌ 클래스 컴포넌트 방식: useEffect 내에서 라우터를 직접 조작
function useRouter() {
  const [pathname, setPathname] = useState(window.location.pathname);
  const [query, setQuery] = useState(
    Object.fromEntries(new URLSearchParams(window.location.search)),
  );

  useEffect(() => {
    const handleRouteChange = () => {
      setPathname(window.location.pathname);
      setQuery(Object.fromEntries(new URLSearchParams(window.location.search)));
    };
    // popstate 이벤트로 URL 변화 감지
    window.addEventListener("popstate", handleRouteChange);
    return () => window.removeEventListener("popstate", handleRouteChange);
  }, []);

  return { pathname, query, push };
}
```

`useRouter()`를 직접 구현할 때 `popstate` 이벤트만 사용하면 `pushState`, `replaceState`의 URL 변화는 감지 못하는 한계가 있습니다. 외부 라이브러리 훅의 내부 동작 원리를 숙지해야 이런 실수를 예방할 수 있습니다.

### 13.4.3 훅의 인수는 불변성을 가지고 있어야 한다

리액트의 성능 최적화는 대부분 얕은 비교에 의존합니다. 즉, 객체나 배열 전체를 비교하는 것이 아니라, 현재 렌더링 프롭스나 상태의 참조가 동일한지 확인합니다. 리액트는 인수가 변경되지 않았다고 가정하고 최적화를 수행하기 때문에,
인수를 직접 변경하면 **예측 불가능한 버그**가 발생할 수 있습니다.

**잘못된 예** — Context를 잘못 활용:

```js
// ❌ rabbit이 isSpecial이 아닐 때도 Context를 구독하게 됨
function useGoldenRabbitBad(rabbit: Rabbit): Rabbit {
  const theme = useContext(ThemeContext);

  if (rabbit.isSpecial) {
    rabbit.color = theme.golden;
    rabbit.glow = true;
  }
  return rabbit;
}
```

**올바른 예** — 조건 분기를 훅 밖으로:

```js
// ✅ isSpecial 조건을 먼저 확인 후 필요할 때만 Context 접근
function useGoldenRabbitGood(rabbit: Rabbit): Rabbit {
  const theme = useContext(ThemeContext);

  // ① rabbit이 special인지 먼저 확인
  if (!rabbit.isSpecial) {
    return rabbit; // 불필요한 Context 구독 없이 조기 반환
  }

  // ② isSpecial일 때만 color, glow 속성 변경
  rabbit.color = theme.golden;
  rabbit.glow = true;
  return rabbit;
}
```

함수 컴포넌트는 렌더링마다 다시 실행됩니다. 클래스 컴포넌트에서 `calculateSomething()`처럼 인스턴스 메서드로 분리하던 로직을, 함수형에서는 훅의 규칙을 지키면서 올바르게 배치해야 합니다.
