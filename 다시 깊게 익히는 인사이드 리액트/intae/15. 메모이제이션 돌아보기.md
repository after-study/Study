# Chapter 15 — 리액트 메모이제이션 돌아보기

## 도입

리액트 애플리케이션의 규모가 커질수록 **불필요한 리렌더링과 연산 비용**은 사용자 경험을 해치는 주된 원인이 됩니다. 이 장에서는 성능 문제를 해결하는 핵심 전략인 메모이제이션을 다룹니다.

- 컴포넌트 자체를 메모이제이션하는 `React.memo()`
- 컴포넌트 내부 값을 캐싱하는 `useMemo()`
- 함수의 참조를 안정화하는 `useCallback()`
- 마지막으로, 이 모든 것을 자동화하는 **리액트 컴파일러**

각 도구의 사용법뿐 아니라 **언제, 왜, 어떤 비용으로** 사용해야 하는지를 함께 살펴보겠습니다.

---

## 15.1 리액트 메모이제이션을 돌아봐야 하는 이유

성능 최적화라는 단어를 들으면 많은 개발자가 `memo()`, `useCallback()`, `useMemo()`를 마법 주문처럼 떠올립니다. "느려졌다면 `memo()`로 감싸세요", "모든 함수는 `useCallback()`으로 감싸는 게 좋지 않나요?" 같은 말을 한 번쯤 들어보셨을 겁니다.

하지만 무분별한 최적화에 대한 맹신은 의도치 않은 부작용을 낳고, 심지어 성능을 악화시키기도 합니다.

- 모든 곳에 `useCallback()`을 적용했지만 의존성 배열을 잘못 관리해 함수 재생성을 막지 못했던 경험
- 복잡한 계산을 `useMemo()`로 캐싱했지만 그 계산 자체가 거의 일어나지 않아 메모리만 낭비했던 경험
- `memo()`를 적용했음에도 자식 컴포넌트가 계속 리렌더링되어 골머리를 앓았던 경험

이런 문제들은 메모이제이션을 **'어떻게 쓰는가'만 알 뿐, '왜 쓰는가'와 '얼마의 비용이 드는가'를 깊이 이해하지 못했을 때** 발생합니다.

> **메모이제이션은 공짜가 아닙니다.** 값을 비교하고 이전 값을 기억하기 위한 메모리와 연산 비용이 분명히 수반됩니다. 따라서 **최적화의 이점이 이 비용을 넘어설 때만 사용**하는 것이 현명합니다.

리액트 컴파일러가 등장하면서 이런 고민이 무의미해질 거라 생각할 수도 있습니다. 하지만 컴파일러가 모든 상황에서 만능은 아닙니다. 컴파일러가 무엇을 최적화하고 무엇을 못 하는지 이해하려면, 결국 메모이제이션의 핵심 개념을 알아야 합니다.

## 15.2 memo()를 사용한 렌더링 최적화

부모 컴포넌트가 리렌더링되면 자식 컴포넌트도 **프롭스 변경 여부와 관계없이 항상 리렌더링**됩니다. 이는 리액트의 기본 동작이지만, 자식의 프롭스가 동일하다면 이 렌더링은 불필요한 작업입니다.

`memo()`는 컴포넌트를 감싸 **프롭스를 비교하고, 변경되지 않았다면 이전 렌더링 결과를 재사용**하는 고차 컴포넌트입니다.

> - `Parent → Child`: 부모가 리렌더링되면 자식은 프롭스 변경과 관계없이 항상 리렌더링됨
> - `Parent → React.memo(Child)`: 프롭스가 변경되지 않으면 리렌더링하지 않음

### 15.2.1 memo() 사용법

```tsx
React.memo(Component, arePropsEqual?)
```

- 두 번째 인자(`arePropsEqual`)를 생략하면 `memo()`는 **얕은 비교**를 수행합니다. `prevProps`와 `nextProps`의 각 속성을 `Object.is()`로 비교하는 방식입니다.
- 직접 비교 함수를 제공하면 `true` 반환 시 리렌더링을 생략하고, `false` 반환 시 리렌더링을 수행합니다. 깊은 비교나 특정 프롭스만 비교해야 할 때 유용합니다.

```tsx
// 일반 컴포넌트: 부모가 리렌더링될 때마다 함께 리렌더링됨
const TitleComponent = ({ title }: { title: string }) => {
  console.log(`${title} 렌더링`);
  return <h4>{title}</h4>;
};

// 메모이제이션된 컴포넌트: title이 같으면 리렌더링되지 않음
const MemoizedTitleComponent = React.memo(({ title }: { title: string }) => {
  console.log(`${title} (메모이즈) 렌더링`);
  return <h4>{title}</h4>;
});

export const MemoExample = () => {
  const [count, setCount] = useState(0);
  return (
    <div>
      <button onClick={() => setCount((c) => c + 1)}>카운트: {count}</button>
      <TitleComponent title="일반 타이틀" />
      <MemoizedTitleComponent title="메모이즈된 타이틀" />
    </div>
  );
};
```

버튼을 클릭하면 `count`가 변해 부모는 리렌더링되지만, `MemoizedTitleComponent`는 `title`이 변하지 않았기에 첫 렌더링 이후로는 리렌더링되지 않습니다.

### 15.2.2 성급한 최적화의 함정

`memo()`가 유용해 보인다고 해서 모든 컴포넌트에 무분별하게 적용하는 것은 좋지 않습니다. **프롭스 비교에도 비용이 들기 때문**입니다. 다음 기준을 적용해보시기 바랍니다.

1. **컴포넌트가 순수하고, 렌더링 비용이 비싸다**: 복잡한 계산을 수행하거나 무거운 UI를 렌더링한다면 효과적입니다.
2. **동일한 프롭스로 자주 리렌더링된다**: 부모의 상태 변화가 자식과 무관한데도 자주 리렌더링될 때 유용합니다.

예를 들어 매 렌더링마다 캔버스에 랜덤 노이즈를 그리는 `NoiseCanvas` 같은 컴포넌트가 있다고 합시다.

```tsx
useEffect(() => {
  const canvas = canvasRef.current;
  if (!canvas) return;
  const ctx = canvas.getContext("2d");
  if (!ctx) return;
  const imageData = ctx.createImageData(width, height);
  for (let i = 0; i < imageData.data.length; i += 4) {
    imageData.data[i] = Math.random() * 255;     // R
    imageData.data[i + 1] = Math.random() * 255; // G
    imageData.data[i + 2] = Math.random() * 255; // B
    imageData.data[i + 3] = 255;                 // A
  }
  ctx.putImageData(imageData, 0, 0);
}); // 매 렌더링마다 실행

const MemoizedNoiseCanvas = React.memo(NoiseCanvas);
```

부모의 상태 변경이 노이즈 이미지와 무관하다면, 매번 픽셀을 새로 그리는 비용은 낭비입니다. `React.memo()`로 감싸면 이 무거운 연산이 불필요하게 반복되지 않습니다.

**판단이 어렵다면 프로파일링 도구로 병목을 확인한 후 필요한 곳에만 적용**하는 것이 현명한 접근입니다.

### 15.2.3 ⚠️ 중첩 구조와 children 프롭스의 함정

`memo()`를 부모와 자식 모두에게 적용했는데도 부모가 리렌더링되는 상황이 있습니다. 특히 `children` 프롭스를 쓸 때 자주 발생합니다.

```tsx
const MemoizedParent = React.memo(({ children }: { children: React.ReactNode }) => {
  console.log("부모 컴포넌트 렌더링");
  return <div>{children}</div>;
});

const MemoizedChild = React.memo(() => {
  console.log("자식 컴포넌트 렌더링");
  return <div>자식 컴포넌트</div>;
});

export default function MemoChildrenDemo() {
  const [count, setCount] = useState(0);
  return (
    <div>
      <button onClick={() => setCount((c) => c + 1)}>count: {count}</button>
      <MemoizedParent>
        <MemoizedChild />
      </MemoizedParent>
    </div>
  );
}
```

`count`가 변해 `MemoChildrenDemo`가 리렌더링될 때, 둘 다 `memo()`이고 프롭스도 없으니 둘 다 리렌더링되지 않을 거라 예상하기 쉽습니다. 하지만 **실제로는 "부모 컴포넌트 렌더링" 로그가 매번 출력**됩니다.

**원인은 `children`의 참조 동일성에 있습니다.** JSX는 내부적으로 다음과 같이 변환됩니다.

```tsx
React.createElement(MemoizedParent, {
  children: React.createElement(MemoizedChild), // 매번 새로운 객체!
});
```

부모(`MemoChildrenDemo`)가 리렌더링될 때마다 `React.createElement(MemoizedChild)`가 새 객체를 만듭니다. `memo()`는 얕은 비교를 하므로, 이전 `children`과 새 `children`의 참조가 다르다고 판단해 `MemoizedParent`를 리렌더링합니다.

> `memo()`는 둘러싸인 컴포넌트가 받는 프롭스가 같으면 리렌더링을 막아주는 역할일 뿐, **`children` 프롭스 자체를 메모이제이션해주지는 않습니다.**

(참고로 `MemoizedChild`가 리렌더링되지 않는 이유는, `MemoizedParent`가 다시 그릴 때 리액트가 "`MemoizedChild`는 `memo()`로 감싸여 있고 프롭스가 동일하다"는 점을 인지하고 렌더링을 생략하기 때문입니다.)

**해결**: `useMemo()`로 `children` 엘리먼트의 참조를 안정화합니다.

```tsx
const memoizedChild = useMemo(() => <MemoizedChild />, []);

return (
  <MemoizedParent>
    {memoizedChild}
  </MemoizedParent>
);
```

> **결론**: `memo()`는 컴포넌트의 **렌더링 결과**를 메모이제이션할 뿐, **전달되는 프롭스를 안정화해주지는 않습니다.** 함수·객체·JSX 엘리먼트를 프롭스로 전달할 때는 `useCallback()`이나 `useMemo()`를 함께 사용해 참조 안정성을 보장해야 `memo()`가 의도대로 동작합니다.

## 15.3 useMemo(): 값 메모이제이션

`memo()`가 컴포넌트의 렌더링 결과를 메모이제이션하는 고차 컴포넌트였다면, `useMemo()`는 **컴포넌트 내부에서 사용하는 특정 값이나 연산 결과를 메모이제이션**하는 훅입니다. 주된 목적은 **렌더링마다 반복할 필요가 없는 비용이 큰 연산을 피하는 것**입니다.

```ts
function useMemo(factory, deps);
```

- `factory`: 메모이제이션할 값을 계산해서 반환하는 함수. 첫 렌더링 시 실행되고 결과가 저장됩니다.
- `deps`: 의존성 배열. 리렌더링 시 이전 값과 현재 값을 `Object.is()`로 비교하여, 하나라도 변경되면 `factory`를 다시 실행해 갱신하고, 모두 같으면 저장된 값을 그대로 반환합니다.

### 15.3.1 useMemo() 내부 동작 원리

리액트의 실제 구현은 복잡하지만, 핵심 원리는 다음과 같이 요약할 수 있습니다.

```ts
function useMemo<T>(callback: () => T, dependencies: any[] | undefined): T {
  // 현재 컴포넌트의 훅 저장소에서 hookIndex 위치의 훅 상태를 가져옴
  const hookIndex = hooks.index++;
  let hook = hooks.list[hookIndex];
  if (!hook) {
    hook = { memoizedValue: null, deps: undefined };
    hooks.list[hookIndex] = hook;
  }

  // 의존성 배열 비교: 메모이제이션의 핵심
  if (!hook.deps || !depsEqual(dependencies, hook.deps)) {
    hook.memoizedValue = callback(); // 새 값 계산
    hook.deps = dependencies;
  }

  return hook.memoizedValue;
}

function depsEqual(newDeps, oldDeps) {
  if (newDeps.length !== oldDeps.length) return false;
  return newDeps.every((dep, i) => Object.is(dep, oldDeps[i]));
}
```

여기서 두 가지 중요한 사실이 드러납니다.

> 1. **상태는 컴포넌트 외부에 저장됩니다** — 리액트는 내부 저장소에 각 훅의 `memoizedValue`와 `deps`를 보관합니다. 덕분에 함수 컴포넌트가 리렌더링되어도 이전 상태를 기억할 수 있습니다.
> 2. **메모이제이션의 핵심은 의존성 비교입니다** — `Object.is()` 기반 얕은 비교로 이전과 다를 때만 새 값을 계산합니다.

또한 훅이 항상 같은 순서로 호출되어야 하는 이유도 여기서 드러납니다. 리액트는 `hookIndex`로 각 훅의 상태를 매칭하기 때문에, 조건문 안에서 훅을 호출하면 순서가 어긋나 잘못된 상태가 매핑됩니다.

### 15.3.2 useMemo()를 활용한 연산 최적화 예제

피보나치 수열은 `n`이 커질수록 연산량이 기하급수적으로 증가하는 대표적인 무거운 연산입니다.

```tsx
function calculateFibonacci(n: number): number {
  if (n <= 1) return 1;
  return calculateFibonacci(n - 1) + calculateFibonacci(n - 2);
}

function generateComplexPattern(seed: number, size: number): string[] {
  let pattern: string[] = [];
  for (let i = 0; i < size; i++) {
    let value = calculateFibonacci(seed + (i % 10));
    pattern.push(`hsl(${value % 360}, 70%, 50%)`);
  }
  return pattern;
}
```

이 함수를 쓰는 컴포넌트가 `seed`와 무관한 `count` 상태를 같이 가진다고 합시다.

```tsx
// ❌ 최적화 전: count 변경에도 비싼 연산이 매번 재실행됨
function UnoptimizedRabbitPattern() {
  const [seed, setSeed] = useState(10);
  const [count, setCount] = useState(0);

  // 매 렌더링마다 generateComplexPattern() 호출 → UI 멈춤
  const colors = generateComplexPattern(seed, 20);

  return (/* ... */);
}
```

`count` 버튼을 누르면 `seed`와 상관없이 매번 무거운 연산이 다시 실행되어 UI가 잠시 멈춥니다. 이를 `useMemo()`로 해결합니다.

```tsx
// ✅ 최적화 후: seed가 바뀔 때만 재실행
function OptimizedRabbitPattern() {
  const [seed, setSeed] = useState(10);
  const [count, setCount] = useState(0);

  const colors = useMemo(() => {
    return generateComplexPattern(seed, 20);
  }, [seed]);

  return (/* ... */);
}
```

이제 `count` 변경 시 `useMemo()`는 캐시된 `colors`를 즉시 반환합니다.

> **테스트 결과** (seed: 25 기준)
> - 최적화 전 `count` 버튼: 화면 업데이트까지 약 9초
> - 최적화 후 `count` 버튼: 즉시 업데이트

## 15.4 useCallback(): 함수 메모이제이션

`useMemo()`가 함수의 **계산 결과**를 메모이제이션했다면, `useCallback()`은 **함수 그 자체**를 메모이제이션하는 훅입니다.

```ts
function useCallback(callback, deps);
```

사실 `useMemo()`로도 동일한 기능을 표현할 수 있습니다.

```ts
// 두 코드는 완전히 동일하게 동작합니다
const memoizedFn  = useCallback(fn, deps);
const memoizedFn2 = useMemo(() => fn, deps);
```

가독성을 위해 함수를 메모이제이션할 때는 `useCallback()`을 씁니다.

### 15.4.1 커스텀 훅에 안정적인 콜백 전달하기

Escape 키로 모달을 닫는 `useEscapeKey` 커스텀 훅이 있다고 합시다. 이 훅은 내부에서 `useEffect`의 의존성 배열에 `onClose`를 포함합니다.

```tsx
// ❌ useCallback 없음 → count 변경 시마다 handleClose가 새 함수로 재생성
function ModalWithoutCallback() {
  const [isOpen, setIsOpen] = useState(false);
  const [count, setCount] = useState(0); // 모달과 무관한 상태

  const handleClose = () => {
    console.log("모달 닫기");
    setIsOpen(false);
  };

  useEscapeKey(handleClose);
  // ...
}
```

`count`가 변할 때마다 `handleClose`가 새 참조로 생성되고, `useEscapeKey` 내부의 `useEffect`는 매번 이벤트 리스너를 제거·재등록합니다. 매우 비효율적입니다.

```tsx
// ✅ useCallback으로 참조 고정
function ModalWithCallback() {
  const [isOpen, setIsOpen] = useState(false);
  const [count, setCount] = useState(0);

  const handleClose = useCallback(() => {
    console.log("모달 닫기");
    setIsOpen(false);
  }, []);

  useEscapeKey(handleClose);
  // ...
}
```

이제 `count`가 변해도 `handleClose`의 참조는 동일하므로 이벤트 리스너가 재등록되지 않습니다.

### 15.4.2 useCallback()으로 프롭스 메모이제이션하기

`memo()`로 자식을 감싸도, **부모가 함수를 프롭스로 전달하면 매 렌더링마다 새 함수가 만들어져 `memo()`가 무용지물**이 됩니다.

```tsx
const ParentComponentForMemo = () => {
  // ✅ 컴포넌트 생애 동안 같은 참조 유지
  const memoizedHandleClick = useCallback(() => {
    console.log("메모이즈된 핸들러");
  }, []);

  // ❌ 부모가 리렌더링될 때마다 새로 생성됨
  const unmemoizedHandleClick = () => {
    console.log("일반 핸들러");
  };

  return (
    <div>
      {/* 자식 리렌더링되지 않음 */}
      <MemoizedChild onClick={memoizedHandleClick} />

      {/* 부모가 리렌더링될 때마다 자식도 리렌더링됨 */}
      <MemoizedChild onClick={unmemoizedHandleClick} />
    </div>
  );
};
```

### 15.4.3 ⚠️ useCallback() 활용: 디바운스 올바르게 동작하게 하기

`useCallback()`이 진짜 필수가 되는 대표적인 케이스가 **디바운스/쓰로틀처럼 내부 상태(타이머 ID)를 가진 함수**입니다.

검색어를 입력할 때마다 API를 호출한다고 합시다. 사용자가 'love'를 입력하면 'l', 'lo', 'lov', 'love'에 대해 각각 API가 호출되어 불필요한 네트워크 부담이 생기고, 응답 순서가 뒤바뀌어 'lo'의 응답이 'love'보다 늦게 도착하는 **경합 조건(race condition)**까지 발생할 수 있습니다.

이를 해결하는 디바운스 기법은 다음과 같이 동작합니다:

> 1. 함수가 호출되면 300ms 타이머를 설정합니다.
> 2. 시간 내에 같은 함수가 다시 호출되면 기존 타이머를 취소하고 새 타이머를 설정합니다.
> 3. 타이머가 만료될 때까지 추가 호출이 없으면, 마지막 호출만 실행합니다.

핵심은 **`debounce()` 함수가 자신의 내부 상태(타이머 ID)를 유지하기 때문에 '기존 타이머 취소'가 가능하다**는 점입니다.

```tsx
import { debounce } from "lodash-es";

const QuoteSearchComponent = () => {
  const [searchTerm, setSearchTerm] = useState("");
  const [results, setResults] = useState<Quote[]>([]);

  const debouncedSearch = useCallback(
    debounce(async (query: string) => {
      if (query.trim() === "") {
        setResults([]);
        return;
      }
      const response = await fetch(`/quote?query=${encodeURIComponent(query)}`);
      const data = await response.json();
      setResults(data.results || []);
    }, 300),
    [], // ✅ 참조를 고정해 동일한 디바운스 함수가 유지되도록 함
  );

  const handleChange = (e: SyntheticEvent<HTMLInputElement>) => {
    setSearchTerm(e.currentTarget.value);
    debouncedSearch(e.currentTarget.value);
  };

  return (
    <div>
      <input type="text" value={searchTerm} onChange={handleChange} />
      <MemoizedQuoteList quotes={results} />
    </div>
  );
};
```

**`useCallback()` 없이 쓰면 왜 디바운스가 동작하지 않는가?**

1. 사용자가 'l'을 입력 → 디바운스된 함수 A 호출 → A는 내부적으로 300ms 타이머 설정
2. 사용자가 'o'를 빠르게 입력 → 컴포넌트가 리렌더링되고, `debounce(...)`가 다시 실행되어 **완전히 새로운 디바운스 함수 B**가 생성됨
3. 함수 B는 함수 A의 타이머 존재를 **전혀 알지 못합니다**. 그래서 A의 타이머를 취소하지 못하고 자신의 새 타이머를 설정합니다
4. 결과적으로 A의 타이머와 B의 타이머가 **각각 독립적으로 만료되어 API가 여러 번 호출**됩니다

컴포넌트가 리렌더링될 때마다 `debounce(...)` 호출이 새 함수를 만들고, 그 안의 타이머도 매번 다른 객체가 됩니다. `useCallback()`을 빈 의존성 배열과 함께 사용하면 **컴포넌트 생애 동안 같은 디바운스 함수를 재사용**해, 내부 타이머 ID도 일관되게 유지됩니다.

이처럼 `useCallback()`은 단순히 함수 재생성을 막는 것을 넘어, **디바운스처럼 내부 상태를 가지며 호출 빈도를 제어하는 함수에서는 정확한 동작을 위한 필수 도구**입니다.

## 15.5 리액트 컴파일러를 이용한 자동 메모이제이션

지금까지 살펴본 `memo()`, `useMemo()`, `useCallback()`은 강력한 도구이지만 **수동 메모이제이션**이라는 한계를 가지고 있습니다. 리액트 컴파일러는 이 한계를 해결하기 위해 등장했습니다.

### 15.5.1 수동 메모이제이션의 한계

1. **코드 복잡성 증가**: 순수한 UI를 표현하는 코드에 최적화 코드가 섞이면서 가독성이 떨어집니다.
2. **실수하기 쉬움**: 메모이제이션 적용을 잊거나 잘못 적용하는 일이 자주 발생합니다.
3. **유지보수의 어려움**: 코드가 변경될 때마다 의존성 배열도 함께 관리해야 합니다.

수동 메모이제이션은 '성능 개선'이라는 이점을 얻기 위해 가독성·유지보수성·작성 단순함을 모두 양보하지 않고 해내야 하는, 매우 까다로운 작업입니다. 리액트팀은 **개발자의 노력 없이 컴파일러가 자동으로 분석해서 최적화를 수행**하는 것을 목표로 삼았습니다.

### 15.5.2 컴파일러의 역할과 기능

리액트 컴파일러는 컴포넌트와 훅 코드를 분석해 다음을 자동으로 수행합니다.

1. **자동 메모이제이션**: 프롭스, 컴포넌트 내부 값, 객체, 함수들을 자동으로 메모이제이션합니다. `useMemo()`, `useCallback()`을 일일이 작성하지 않아도 같은 효과를 얻습니다.
2. **캐스케이딩 렌더링 방지**: 부모가 리렌더링될 때 자식도 자동으로 메모이제이션되어, 불필요한 연쇄 리렌더링을 막습니다.
3. **코드 가독성 향상**: `React.memo()`, `useMemo()`, `useCallback()` 호출이 대부분 사라져 개발자가 UI를 선언적으로 표현하는 일에 집중할 수 있습니다.

### 15.5.3 동작 원리

리액트 컴파일러는 **리액트의 규칙을 준수한다는 전제** 하에 빌드 시점에 정적으로 코드를 분석해 최적화를 수행합니다.

1. **컴포넌트는 멱등적이어야 합니다**: 같은 프롭스에 대해 항상 같은 결과를 반환해야 합니다.
2. **컴포넌트는 순수 함수여야 합니다**: 렌더링 중에 부수 효과를 발생시키면 안 됩니다.
3. **훅 규칙을 준수해야 합니다**: 훅은 컴포넌트나 커스텀 훅의 최상위에서 호출해야 합니다.

다음은 간단한 `ProductItem` 컴포넌트가 컴파일러를 거치면 어떻게 변환되는지 보여줍니다.

```tsx
// 원본
const ProductItem = ({ product, onAddToCart }: ProductItemProps) => {
  return (
    <div>
      <p><strong>{product.name}</strong> - {product.price.toLocaleString()}원</p>
      <button onClick={() => onAddToCart(product.id)}>장바구니 추가</button>
    </div>
  );
};
```

컴파일러는 이 코드를 슬롯 기반 캐시를 활용하는 형태로 변환합니다.

```js
import { c as _c } from "react/compiler-runtime";

const ProductItem = (t0) => {
  const $ = _c(13); // ❶ 13개 슬롯의 캐시 배열 생성
  const { product, onAddToCart } = t0;

  let t1;
  if ($[0] !== product.name) {
    // ❷ product.name이 이전과 다르면 JSX 새로 생성
    t1 = <strong>{product.name}</strong>;
    $[0] = product.name; // 현재 값 저장
    $[1] = t1;           // 생성된 JSX 저장
  } else {
    t1 = $[1]; // 같으면 캐시된 JSX 재사용
  }

  let t2;
  if ($[2] !== product.price) {
    t2 = product.price.toLocaleString();
    $[2] = product.price;
    $[3] = t2;
  } else {
    t2 = $[3];
  }
  // ...
};
```

- ❶ `react-compiler-runtime`의 `_c()`로 메모이제이션 캐시 배열을 만듭니다. 컴포넌트가 호출되는 시점에 컴포넌트별로 생성됩니다.
- ❷ JSX 엘리먼트·객체 참조·함수 참조 등 값마다 이전 값과 비교해, 변경되지 않았다면 캐시된 값을 재사용합니다.

`product.name`이 변경되지 않으면 `<strong>` 엘리먼트는 다시 생성되지 않고 재사용됩니다.

### 15.5.4 리액트 컴파일러는 수동 메모이제이션을 완전히 대체하는가?

**아닙니다.** 상황에 따라 개발자가 직접 메모이제이션해야 하는 경우가 여전히 있습니다.

예를 들어 다음과 같은 `ProductList`를 살펴봅시다.

```tsx
function ProductList({ products, searchTerm, onAddToCart }) {
  const filteredProducts = products.filter((product) =>
    product.name.toLowerCase().includes(searchTerm.toLowerCase())
  );

  if (filteredProducts.length === 0 && searchTerm) {
    return <p>"{searchTerm}"에 해당하는 제품이 없습니다.</p>;
  }

  return (
    <div>
      {filteredProducts.map((product) => (
        <ProductItem key={product.id} product={product} onAddToCart={onAddToCart} />
      ))}
    </div>
  );
}
```

컴파일러는 `products`나 `searchTerm`이 변경되지 않으면 `filter` 결과를 재사용하고, 변경되지 않은 JSX 블록도 캐시해 재사용합니다. 한 줄도 바꾸지 않은 코드에서 자동으로 의존성별 캐싱이 일어나는 것이 핵심입니다.

하지만 **`onAddToCart` 같은 함수 프롭스가 외부에서 매번 새 참조로 들어온다면**, 컴파일러가 만든 캐시도 자주 무효화됩니다. 이런 경우에는 여전히 부모 쪽에서 `useCallback()`으로 함수 참조를 안정화하는 수동 메모이제이션이 필요합니다.

즉, 컴파일러는 강력한 자동화 도구지만 **외부에서 흘러들어오는 참조의 안정성까지 보장하지는 못합니다.** 메모이제이션의 핵심 개념을 알고 있어야 컴파일러가 무엇을 하고 무엇을 못 하는지 판단할 수 있습니다.

## 핵심 정리

리액트 컴파일러는 리액트 19와 함께 발표되었지만 리액트 자체에 내장되어 있지 않고, 별도의 바벨 플러그인으로 적용해야 합니다. 그리고 컴파일러가 있더라도 다음 개념을 이해해야 컴파일러의 한계와 동작을 제대로 다룰 수 있습니다.

- **`memo()`** — 컴포넌트의 렌더링 결과를 메모이제이션하는 고차 컴포넌트입니다. 다만 **전달되는 프롭스 자체를 안정화해주지는 않으므로**, `children`·함수·객체 프롭스는 별도로 `useMemo()`·`useCallback()`으로 안정화해야 합니다.
- **`useMemo()`** — 비싼 연산의 결과 값을 메모이제이션해, 매 렌더링마다 불필요하게 재계산되는 일을 막습니다.
- **`useCallback()`** — 함수의 참조 안정성을 보장합니다. `memo()`된 자식에 함수 프롭스를 전달할 때, 그리고 **디바운스·쓰로틀처럼 내부 상태를 가진 함수에서는 정확한 동작을 위한 필수 도구**입니다.
- **리액트 컴파일러** — 수동 메모이제이션을 자동화하지만 만능은 아닙니다. 외부에서 들어오는 참조의 안정성까지는 보장하지 못하므로, 메모이제이션의 핵심 개념을 이해하고 있어야 합니다.
