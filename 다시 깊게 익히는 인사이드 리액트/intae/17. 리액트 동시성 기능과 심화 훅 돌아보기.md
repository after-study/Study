# Chapter 17 — 리액트 동시성 기능과 심화 훅 돌아보기

## 도입

리액트 개발의 중심 기능은 여전히 `useState()`, `useEffect()`이지만, 이것만으로는 해결하기 어려운 문제들이 존재합니다. 외부 라이브러리와 DOM을 직접 다룰 때 발생하는 미세한 깜빡임(flickering), 무거운 작업으로 인한 입력 지연, 외부 스토어와의 동기화 과정에서 발생하는 테어링(tearing) 같은 문제들이 그것입니다.

이 장에서는 다음과 같은 심화 훅과 동시성 기능을 다룹니다.

- `useLayoutEffect()` / `useInsertionEffect()` — DOM 변경 이전에 동기적으로 실행되는 훅
- `useImperativeHandle()` — 부모가 자식 컴포넌트를 명령형으로 제어하는 방법
- `useTransition()` / `useDeferredValue()` — 리액트 18에서 도입된 동시성 기능
- `useSyncExternalStore()` — 외부 상태를 안전하게 동기화하는 훅

각 훅이 **왜 필요했는지**, **언제 써야 하는지**를 깊이 있게 살펴봅니다.

---

## 17.1 리액트 동시성 기능과 심화 훅을 돌아봐야 하는 이유

리액트 18버전부터 도입된 동시성 기능은 단순히 새로운 API가 추가된 것을 넘어 **렌더링의 우선순위를 제어할 수 있게** 했습니다. 더 이상 모든 상태 업데이트가 동등한 우선순위로 처리되지 않습니다.

- `useTransition()` / `useDeferredValue()`: 긴급한 업데이트와 그렇지 않은 업데이트를 구분
- `useSyncExternalStore()`: 외부 상태와 리액트의 동시성 렌더링 사이의 일관성 보장
- `useLayoutEffect()` / `useInsertionEffect()` / `useImperativeHandle()`: 정밀한 렌더링 제어와 명령형 API 노출

이러한 훅들은 "쓰지 않아도 앱은 동작하지만, 쓰면 사용자 경험을 한 단계 끌어올릴 수 있는" 도구들입니다. **언제 필요한지 판단할 수 있어야** 진정한 가치를 발휘합니다.

## 17.2 useLayoutEffect()와 useInsertionEffect() 사용 시점

### 17.2.1 useEffect()와 useLayoutEffect()의 차이점

가장 흔한 질문 중 하나는 "useEffect와 useLayoutEffect 중 무엇을 써야 하는가?" 입니다. 두 훅은 비슷해 보이지만 **실행 시점이 결정적으로 다릅니다.**

| 훅 | 실행 시점 | 특징 |
|---|---|---|
| `useEffect()` | 브라우저가 화면을 그린 **후** 비동기적으로 실행 | 렌더링을 차단하지 않음 |
| `useLayoutEffect()` | DOM 변경 후, 브라우저가 화면을 그리기 **전에** 동기적으로 실행 | 렌더링을 차단함 |

**문제 상황: Chart.js로 차트를 그리는 컴포넌트**

```tsx
// ch17/src/components/CarrotPriceChartEffect.tsx
const CarrotPriceChart = ({ data }: { data: CarrotData }) => {
  const chartRef = useRef<Chart | null>(null);
  const canvasRef = useRef<HTMLCanvasElement | null>(null);

  useEffect(() => {
    // 1. 이전 차트 인스턴스 파괴 (메모리 누수 방지)
    if (chartRef.current) {
      chartRef.current.destroy();
    }
    // 2. canvas 요소로부터 2D 컨텍스트 가져옴
    const ctx = canvasRef.current!.getContext('2d') as ChartItem;
    // 3. Chart.js로 새로운 차트 인스턴스 생성
    chartRef.current = new Chart(ctx, {
      type: 'line',
      data: getChartData(data),
    });

    // 4. 클린업 함수
    return () => {
      if (chartRef.current) {
        chartRef.current.destroy();
      }
    };
  }, [data]);

  return (
    <div style={{ width: '900px', height: '400px' }}>
      <canvas ref={canvasRef} />
    </div>
  );
};
```

이 코드는 잘 동작하는 것처럼 보이지만, 자세히 보면 **차트가 한 번 깜빡인 뒤에 다시 그려지는 미세한 깜빡임 현상**이 발생합니다. 이유는 다음과 같습니다.

1. **렌더링** — 빈 캔버스가 화면에 페인트됨
2. **useEffect 실행** — 차트가 그려짐
3. **재페인트** — 새로 그려진 차트가 화면에 나타남

`useLayoutEffect()`로 바꾸면 빈 캔버스가 페인트되기 전에 차트가 먼저 그려지므로 깜빡임이 사라집니다.

> **결론**: DOM을 측정하거나 직접 조작해서 화면에 그려야 하는 작업은 `useLayoutEffect()`를 사용하세요. 단, `useLayoutEffect()`는 **렌더링을 차단하기 때문에 성능 저하의 원인이 될 수 있습니다.** 동기적 실행이 꼭 필요한 경우에만 사용해야 합니다.

### 17.2.2 리액트 헬멧과 useInsertionEffect()

`react-helmet`은 컴포넌트 내부에서 `<head>`의 메타데이터(`<title>`, `<meta>`, `<style>`, `<script>`)를 선언할 수 있게 해주는 라이브러리입니다. 리액트가 자체적으로 페이지의 head를 관리하지 않기에 SEO 최적화 및 메타 정보 관리를 위해 자주 사용됐습니다.

```tsx
// ch17/src/components/CarrotPriceChartEffect.tsx
import { Helmet } from "react-helmet";

return (
  <div>
    <Helmet>
      <title>당근 가격 차트</title>
      <meta name="description" content="annual carrot price explain" />
    </Helmet>
    <canvas ref={canvasRef} />
  </div>
);
```

`useInsertionEffect()`는 **DOM 변경이 일어나기 전에 실행**되어, CSS-in-JS 라이브러리가 스타일 태그를 주입할 때 사용됩니다.

```tsx
// ch17/src/components/CarrotPriceChartEffectInjector
const DynamicStyleInjector: React.FC<DynamicStyleInjectorProps> = ({ rule }) => {
  const styleIdSuffix = useId();
  const styleElementId = `dynamic-style-${styleIdSuffix}`;

  useInsertionEffect(() => {
    // 1. 기존에 동일 ID로 생성된 스타일 태그가 있다면 제거
    const oldStyleElement = document.getElementById(styleElementId);
    if (oldStyleElement) {
      oldStyleElement.remove();
    }

    // 2. 새로운 <style> 태그 생성
    const styleElement = document.createElement('style');
    styleElement.id = styleElementId;
    styleElement.innerHTML = rule;

    // 3. document.head에 <style> 태그 추가
    document.head.appendChild(styleElement);

    // 4. 컴포넌트 언마운트 시 또는 rule이 변경되어 이펙트가 재실행되기 직전에 호출
    return () => {
      const styleTagToRemove = document.getElementById(styleElementId);
      if (styleTagToRemove) {
        document.head.removeChild(styleTagToRemove);
      }
    };
  }, [rule]);

  return null;
};
```

### 이펙트 훅의 실행 순서

세 가지 이펙트 훅의 실행 순서를 정확히 이해해야 합니다.

```
컴포넌트 렌더링 → useInsertionEffect() → useLayoutEffect() → useEffect()
       (브라우저 페인트는 useLayoutEffect 이후 / useEffect 이전에 발생)
```

- **`useInsertionEffect()`**: 컴포넌트 마운트 후 → 레이아웃 계산 전 (스타일 주입용)
- **`useLayoutEffect()`**: DOM 변경 후 → 페인트 전 실행 (DOM 측정·조작)
- **`useEffect()`**: 컴포넌트 마운트 후 실행 (일반적인 사이드 이펙트)

### 17.2.3 리액트 19버전의 메타데이터와 스타일시트

리액트 19부터는 `react-helmet` 없이도 컴포넌트 내부에서 `<title>`, `<meta>`, `<link>`를 선언하면 **리액트가 자동으로 `<head>`로 이동시킵니다.**

```tsx
// ch17/src/components/ComponentWithReact19.tsx
function ComponentWithReact19({ product }) {
  return (
    <div>
      <title>{product.name}</title>
      <meta name="description" content={product.description} />
      <meta property="og:title" content={product.name} />
      <link rel="canonical" href={`https://.../products/${product.id}`} />
      <h1>{product.name}</h1>
    </div>
  );
}
```

**`precedence` 프롭스로 스타일 우선순위 제어**

`<link rel="stylesheet">` 사용 시 `precedence` 프롭스를 통해 스타일시트 간 우선순위를 명시할 수 있습니다.

```tsx
<link rel="stylesheet" href="/style-first.css" precedence="foo" />
<link rel="stylesheet" href="/style-third.css" precedence="bar" />
<link rel="stylesheet" href="/style-second.css" precedence="foo" />
```

`precedence`가 같은 그룹은 함께 묶여 작성된 순서대로 head에 영향을 줍니다. 또한 Next.js와 같이 작성한 한정에서 no-css-tags 등의 경고를 받지 않고도 의도된 우선순위를 제어할 수 있습니다.

**Suspense와의 결합**

리액트 19에서 `<link>`로 선언된 외부 스타일시트가 다운로드되기 전까지 컴포넌트가 그려지지 않도록 보장됩니다. 이를 통해 **FOUC(Flash of Unstyled Content)** 이슈를 자연스럽게 피할 수 있습니다.

```tsx
// ch17/src/components/SlowComponent.tsx
const SlowComponent = () => {
  return (
    <div className="slow-component p-6 rounded-lg shadow-lg">
      <link rel="stylesheet" href="/slow.css" precedence="default" />
      <h2 className="text-xl font-bold mb-4 fancy-title">Slow Loading Component</h2>
      <p className="fancy-text">이 텍스트는 CSS가 다운로드될 때까지 보이지 않습니다.</p>
    </div>
  );
};

const App = () => {
  return (
    <Suspense fallback={<Spinner />}>
      <SlowComponent />
    </Suspense>
  );
};
```

## 17.3 useImperativeHandle()을 사용한 제어 역전

리액트는 기본적으로 **선언형(declarative)** 패러다임을 따릅니다. 부모는 자식에게 프롭스를 내려주고, 자식은 그 프롭스를 보고 자기 자신을 렌더링합니다. 하지만 가끔은 **명령형(imperative) 제어가 필요한 순간**이 있습니다.

- 입력 필드의 포커스 주기, 스크롤 위치 제어하기
- 애니메이션을 명령적으로 트리거하기
- 자식 컴포넌트의 특정 액션을 부모가 직접 호출해야 할 때

이런 상황에서 `forwardRef()`만으로는 자식의 내부 DOM 노드를 그대로 노출해야 하는데, 이는 캡슐화를 깨트립니다. `useImperativeHandle()`은 **부모에게 노출할 메서드를 자식이 정의**하는 방식으로 이 문제를 해결합니다.

```tsx
// ch17/src/components/InversionOfControl.tsx
interface ChildInputHandle {
  focusInput: () => void;
  clearInput: () => void;
  getInputValue: () => string | undefined;
}

const ChildComponent = forwardRef<ChildInputHandle, {}>((props, ref) => {
  const inputRef = useRef<HTMLInputElement>(null);

  // 1. useImperativeHandle로 ref가 가리킬 객체(핸들)를 정의
  useImperativeHandle(ref, () => ({
    focusInput: () => {
      inputRef.current?.focus(); // input 요소의 포커스 줌
    },
    getInputValue: () => {
      return inputRef.current?.value; // input의 값을 반환
    },
    clearInput: () => {
      if (inputRef.current) {
        inputRef.current.value = ""; // input 요소의 값을 비움
      }
    },
  }));

  return (
    <div>
      <input type="text" ref={inputRef} />
    </div>
  );
});

const ParentComponent = () => {
  const childRef = useRef<ChildInputHandle>(null);

  return (
    <div>
      <ChildComponent ref={childRef} />
      <button onClick={() => childRef.current?.focusInput()}>포커스</button>
      <button onClick={() => childRef.current?.clearInput()}>초기화</button>
      <button onClick={() => alert(childRef.current?.getInputValue())}>값 가져오기</button>
    </div>
  );
};
```

### useImperativeHandle()의 핵심 특징

- **제어 역전(Inversion of Control)**: 일반적으로 부모가 프롭스를 통해 자식을 제어하지만, `useImperativeHandle()`은 자식이 노출할 인터페이스를 선언하고 부모가 그것만 호출하도록 제한합니다.
- **캡슐화 유지**: 내부 DOM 노드를 직접 노출하지 않고, 안전한 API만 제공함으로써 캡슐화를 유지합니다.

### 언제 써야 하는가?

`useImperativeHandle()`은 강력하지만 **남용은 위험**합니다. 다음 기준이 명확할 때만 사용하시기 바랍니다.

1. **입력 필드의 포커스 주기, 스크롤 위치 제어** 등 DOM 명령적 조작이 필요할 때
2. **애니메이션을 직접 트리거**해야 할 때
3. **자식 컴포넌트의 특정 액션**을 부모가 직접 호출해야 할 때

대부분의 상황은 프롭스와 콜백만으로 해결됩니다. 명령형 API가 정말로 필요한지 다시 한번 검토해보시기 바랍니다.

## 17.4 동시성 기능과 트랜지션

리액트 18버전에서 도입된 동시성(concurrency) 기능은 UI가 무거운 작업 중에 사용자 입력에 즉각 응답하지 못하는 단점을 해결합니다. 핵심 아이디어는 **모든 상태 업데이트가 동등하게 긴급하지 않다**는 것입니다.

- **긴급한 업데이트**: 타이핑, 클릭, 호버 같은 사용자 입력 반응
- **트랜지션 업데이트**: 검색 결과 갱신, 페이지 전환 같은 무거운 작업

리액트는 트랜지션으로 표시된 업데이트를 **잠시 중단하고, 더 긴급한 업데이트를 먼저 처리**할 수 있게 됩니다.

### 17.4.1 useTransition()으로 우선순위 보는 작업 선정하기

`useTransition()`은 특정 상태 업데이트를 트랜지션으로 표시합니다. 수많은 버블을 화면에 표시하는 `BubbleGenerator` 컴포넌트를 예시로 살펴보겠습니다.

```tsx
// ch17/src/components/BubbleGenerator.tsx
const BubbleGenerator = () => {
  // 1. isPending: 트랜지션이 진행 중인지 나타내는 boolean
  //    startTransition: 상태 업데이트를 트랜지션으로 감싸는 함수
  const [isPending, startTransition] = useTransition();

  // 2. listSize에 BubbleList의 실제로 그릴 버블 개수
  const [listSize, setListSize] = useState(1);

  return (
    <>
      {isPending ? (
        <p className="text-indigo-600">트랜지션이 진행 중일 때는 보드한...</p>
      ) : (
        <BubbleList size={listSize} />
      )}
    </>
  );
};
```

#### 트랜지션이 적용되지 않은 경우 vs 적용된 경우

- **트랜지션이 적용되지 않은 경우**: 사용자가 슬라이더를 빠르게 움직이면 매번 모든 버블이 즉시 재계산되어, UI가 멈춘 것처럼 보입니다.
- **트랜지션이 적용된 경우**: 슬라이더는 부드럽게 움직이고, 무거운 버블 렌더링은 잠시 미뤄집니다. 사용자는 응답성을 유지한 채로 작업할 수 있습니다.

> **핵심**: `useTransition()`의 `startTransition()`은 "이 업데이트는 긴급하지 않으니, 더 긴급한 일이 있으면 우선 처리해주세요"라고 리액트에게 알려주는 신호입니다.

### 17.4.2 useDeferredValue()로 지연된 상태값 사용하기

`useTransition()`이 **상태 업데이트의 우선순위를 낮추는 방식**이라면, `useDeferredValue()`는 **이미 존재하는 값의 사용을 지연시키는 방식**입니다.

```tsx
// ch17/src/components/BubbleGenerator.tsx
function BubbleTextInput({ label, value, onChange }) {
  return (
    <div>
      <label>{label}</label>
      <input type="text" value={value} onChange={onChange} />
    </div>
  );
}

const BubbleGenerator = () => {
  // 1. 텍스트 입력 필드를 위한 상태. 즉시 업데이트됨
  const [inputValue, setInputValue] = useState("");

  // 2. inputValue가 지연되어 무거운 렌더링이 즉각적인 입력 반응을
  //    방해하지 않도록 함
  const deferredInputValue = useDeferredValue(inputValue);

  return (
    <>
      <BubbleTextInput
        label="버블에 글자 새기기:"
        value={inputValue}
        onChange={(e) => setInputValue(e.target.value)}
      />
      {/* deferredInputValue가 즉각적인 입력 반응을 막지 않도록 함 */}
      <BubbleList size={listSize} text={deferredInputValue} />
    </>
  );
};
```

**`useTransition()` vs `useDeferredValue()`**

| 구분 | useTransition() | useDeferredValue() |
|---|---|---|
| 제어 위치 | **상태를 업데이트하는 쪽**에서 제어 | **상태를 소비하는 쪽**에서 제어 |
| 적용 대상 | `setState` 호출을 감쌈 | 이미 존재하는 값을 감쌈 |
| 사용 시기 | 상태 변경 코드를 직접 작성할 때 | 외부에서 받은 프롭스를 지연시키고 싶을 때 |

### 17.4.3 startTransition()과 비동기 함수

리액트 18버전에서 `startTransition()`은 콜백 함수의 동작이 유사하나라고 생각한단든 setTimeout과 유사할 것이라고 생각할 수 있습니다.

```tsx
useEffect(() => {
  console.log(1);            // 1
  startTransition(() => {
    console.log(2);          // 2
    setCount(prev => prev + 1);
  });
  console.log(3);            // 3
}, []);
```

`startTransition()` 내부 콜백 함수의 동작이 유사할 따름이며, 실제로는 그 자업이 비동기적으로 처리되는 트랜지션 내부에서 결국 단번에 실행됩니다.

#### 비동기 함수와 트랜지션 (리액트 18 vs 19)

리액트 **18버전**에서는 `startTransition()` 내부에서 비동기 함수의 `await` 이후 코드는 트랜지션 작업으로 처리되지 않습니다. 이를 우회하기 위해 `setIsLoadingManually(true)` 같은 별도의 상태로 진행 상태를 추적해야 했습니다.

```tsx
// ch17/src/components/QuerySearch_React18.ts
const [isPending, startTransition] = useTransition();
const [isLoadingManually, setIsLoadingManually] = useState(false);

const handleSearch = () => {
  startTransition(async () => {
    // 1. React 18: startTransition이 async 함수를 직접 받음
    setIsLoadingManually(true);     // 2. 비동기 API 호출 전 수동 로딩 true
    try {
      const fetchedQuotes = await fetchQuotesAPI(searchTerm);
      setQuotes(fetchedQuotes);     // 3. 비동기 작업 완료 후 상태 업데이트
    } catch (e) {
      setError('검색 결과가 없습니다.');
    } finally {
      setIsLoadingManually(false);  // 4. 비동기 작업 완료 후 수동 로딩 false
    }
  });
};
```

리액트 **19버전**부터는 `startTransition()` 내부에서 `async/await`를 직접 지원합니다. 별도의 수동 로딩 상태 없이 `isPending`만으로 전체 비동기 흐름을 표현할 수 있게 되었습니다.

```tsx
// ch17/src/components/QuerySearch_React19.tsx
const handleSearch = async () => {
  startTransition(async () => {
    // R19: startTransition이 async/await를 직접 지원
    try {
      const fetchedQuotes = await fetchQuotesAPI(searchTerm);
      setQuotes(fetchedQuotes);
    } catch (e) {
      setError('검색 결과가 없습니다.');
    }
  });
};
```

## 17.5 useSyncExternalStore()를 사용한 외부 상태 동기화

### 17.5.1 내부 상태와 외부 상태

리액트의 상태는 두 가지로 나뉩니다.

- **내부 상태(internal state)**: `useState()`, `useReducer()` 등 리액트 컴포넌트 내부에서 관리되는 상태
- **외부 상태(external state)**: 리액트 외부에 존재하는 상태 — Zustand, Jotai, Redux 같은 상태 라이브러리, 브라우저의 전역 변수, `Date`, DOM 요소의 특정 속성값 등

```
┌─────────────────────┐         ┌──────────────────────┐
│   내부 상태          │         │   외부 상태          │
│   useState          │         │   Zustand            │
│   useReducer        │         │   Jotai              │
│                     │         │   Redux              │
│                     │         │   Date / DOM         │
└─────────────────────┘         └──────────────────────┘
```

리액트 컴포넌트가 외부 상태를 구독하려면 이전에는 `useEffect()` 내에서 처리하는 컴포넌트 내부의 useState로 변경하고, 리액트 컴포넌트에서 다시 사용해야 했습니다. 이 과정에서 동시성 모드와 결합되면 **테어링(tearing)** 문제가 발생할 수 있었습니다.

### 17.5.2 useSyncExternalStore()와 테어링

**테어링이란?** 동시성 기능에서 외부 상태가 변경됐을 때, **같은 외부 상태를 참조하는 여러 컴포넌트가 서로 다른 값을 보여주는 시각적 불일치 현상**을 말합니다. 마우스 X 좌표를 추적하는 커스텀 훅 `useMouseX()`를 예시로 살펴봅시다.

```tsx
// ch17/src/components/TearingExampleApp.tsx
const useMouseX = () => {
  const ref = useRef(0); // 마우스 X 좌표를 저장할 ref

  useEffect(() => {
    const handleMouseMove = (event: MouseEvent) => {
      ref.current = event.clientX;
    };
    window.addEventListener('mousemove', handleMouseMove);
    return () => {
      window.removeEventListener('mousemove', handleMouseMove);
    };
  }, []);

  return ref.current; // 현재 마우스 X 좌표 반환
};

const MousePositionX = ({ instanceId }: { instanceId: number }) => {
  // 1 또는 React 19인 버전에서는 동시성 렌더링 중 사용될 경우, 
  //   매 useMouseX 호출이 동일한 시점의 결과를 보장하지 않음
  const x = useMouseX();

  // 의도적인 렌더링 지연 시뮬레이션
  const startTime = performance.now();
  while (performance.now() - startTime < 20) {
    // 20ms 동안 메인 스레드 블로킹
  }

  return <div>Mouse X (Instance {instanceId}): {x}</div>;
};
```

위 코드에서 50개의 `MousePositionX` 컴포넌트를 렌더링하면, 마우스를 움직일 때 **각 컴포넌트가 다른 X 좌표를 표시**하는 현상이 발생합니다. 이것이 바로 테어링입니다. 동시성 렌더링이 중간에 끊겼다가 재개되면서, 그 사이 마우스가 이동한 시점의 새로운 값이 일부 컴포넌트에 적용되기 때문입니다.

#### useSyncExternalStore()로 해결하기

`useSyncExternalStore()`는 외부 스토어를 구독하는 표준 API입니다. **모든 컴포넌트가 같은 시점의 값을 보장**받습니다.

```tsx
// ch17/src/components/NetworkStatus.tsx
const networkStore = {
  isOnline:
    typeof navigator !== 'undefined' ? navigator.onLine : true,
  listeners: new Set<() => void>(),

  // 스토어 상태가 변경되었음을 모든 리스너에게 알리는 함수
  notify() {
    this.listeners.forEach((listener) => listener());
  },

  // 1. 스토어의 현재 상태(스냅샷)를 반환하는 함수
  //    React는 이 함수의 반환값을 사용하여 UI를 렌더링함
  getSnapshot() {
    return this.isOnline;
  },

  // 2. 스토어의 상태 변경을 구독하는 함수
  //    listener: 상태 변경 시 호출될 콜백 함수
  subscribe(listener: () => void) {
    this.listeners.add(listener);

    // 브라우저가 온라인 상태로 변경될 때 실행될 이벤트 리스너 등록
    if (typeof window !== 'undefined') {
      window.addEventListener('online', () => {
        networkStore.isOnline = true;   // 스토어의 isOnline 상태 true로 업데이트
        networkStore.notify();          // 상태 변경을 리스너들에게 알림
      });

      // 브라우저가 오프라인 상태로 변경될 때 실행될 이벤트 리스너 등록
      window.addEventListener('offline', () => {
        networkStore.isOnline = false;  // 스토어의 isOnline 상태 false로 업데이트
        networkStore.notify();
      });
    }

    return () => {
      this.listeners.delete(listener);  // 구독 해지 시 콜백 함수 제거
    };
  },
};

// 인터넷 연결 유무를 표시하는 컴포넌트
const NetworkStatus = () => {
  const isOnline = useSyncExternalStore(
    networkStore.subscribe.bind(networkStore),
    networkStore.getSnapshot.bind(networkStore)
  );

  return (
    <div className={`flex items-center ${isOnline ? 'bg-green-500' : 'bg-red-500'}`}>
      <span>{isOnline ? 'Online' : 'Offline'}</span>
    </div>
  );
};
```

#### useSyncExternalStore()의 두 인자

1. **`subscribe(callback)`**: 외부 스토어에 변경 사항을 알리는 콜백을 등록하는 함수. 구독 해지 함수를 반환해야 함
2. **`getSnapshot()`**: 외부 스토어의 현재 상태 스냅샷을 반환하는 함수. **같은 입력에 대해 같은 출력을 보장**해야 함 (참조 동일성 유지)

> **핵심**: `useSyncExternalStore()`는 리액트의 동시성 렌더링이 외부 상태의 변경을 **원자적으로(atomically)** 인지하도록 보장합니다. 즉, 한 번의 렌더링 사이클 안에서는 모든 컴포넌트가 같은 외부 상태값을 봅니다.

## 마치며

이 장에서 다룬 훅들의 공통점은 **"꼭 필요한 순간이 아니면 쓰지 않는 것이 좋다"**는 것입니다.

| 훅 | 사용 시점 |
|---|---|
| `useLayoutEffect()` | DOM 측정·조작이 페인트 전에 동기적으로 일어나야 할 때 |
| `useInsertionEffect()` | CSS-in-JS 라이브러리에서 스타일 주입 시 |
| `useImperativeHandle()` | 자식의 명령형 API를 부모에게 안전하게 노출해야 할 때 |
| `useTransition()` | 상태 업데이트의 우선순위를 낮추고 싶을 때 |
| `useDeferredValue()` | 받은 값의 사용을 지연시키고 싶을 때 |
| `useSyncExternalStore()` | 외부 상태를 동시성 안전하게 구독해야 할 때 |

리액트의 기본 훅들로 90% 이상의 문제가 해결됩니다. 하지만 나머지 10%에서, 사용자 경험의 질을 결정짓는 순간들을 위해 이 심화 훅들의 존재 이유와 사용법을 알아두어야 합니다.

> **"훅의 종류를 아는 것보다, 각 훅이 해결하는 문제를 아는 것이 중요합니다."**
