# 11 리액트의 상태와 배칭

리액트로 개발하다보면, 객체나 배열을 직접 수정해서, UI가 갱신되지 않는다던가
동기화가 안된다던가의 문제를 경험한적이 있을 것입니다.
그것들은 리액트의 동작원리를 제대로 이해하지 못했을 때 발생하는 일들입니다.
어떻게 리액트가 상태를 업데이트 하는지 알아봅시다.

## 1. 리액트 상태 정의와 종류

리액트 애플리케이션의 동적인 UI는 변화하는 상태에 의해 구동됩니다. 상태는 단순한 변수를 넘어 컴포넌트가 자신을 언제 다시 그려야 할지를 결정하는 핵심 정보입니다.

### 1.1 지역 상태와 파생 상태

컴포넌트 내부에 선언되어 다른 컴포넌트에서는 참조하지 못하는 내부 상태값을 지역 상태라고 이야기합니다. 따라서 컴포넌트 내부에서 useState()로 만든 상탯값은 지역상태입니다.
리액트는 참조하는 상탯값이 변경될 때 마다 새로운 스냅샷을 생성한다!

**스냅샷** : 특정 렌더링 시점의 UI 상태를 캡쳐한 것을 의미함.

- 컴포넌트 함수가 호출될 때, React는 그 시점의 state, props, 이벤트 핸들러 등을 포함한 JSX 결과물(React Element 트리)을 생성합니다. 이것이 스냅샷입니다.
- 핵심은 클로저입니다. 이벤트 핸들러나 effect 내부에서 참조하는 state 값은 해당 렌더링 시점에 캡처된 값이지, 최신 값이 아닙니다.

### 1.2 상태의 불변성

리액트애서 상태는 불변하게 다뤄야합니다. 불변성이란 데이터를 직접 변경하는 것이 아닌 항상 새로운 데이터로 교체하는 방식으로 변경을 처리하는 것을 말합니다.
리액트는 객체의 참조를 비교하여 상태 변화의 여부를 판단하기 때문에, 직접 비교하면 리액트는 불필요한 렌더링을 방지하기 위해 상태 변화를 생략할 수 있습니다.

```javascript
function dispatchSetStateInternal(fiber, queue, action) {
  // 현재 렌더링 중이 아니고, 큐가 비어있고, 다른 업데이트도 없는 특별한 경우에만 진입
  if (isOptimizablePath(fiber, queue)) {
    // 이전에 렌더링된 리듀서와 상태를 가져옴
    lastRenderedReducer = queue.lastRenderedReducer;
    currentState = queue.lastRenderedState;

    if (lastRenderedReducer) {
      // 새로운 상태를 미리 계산 (eager computation)
      eagerState = lastRenderedReducer(currentState, action);

      // Object.is를 사용하여 미리 계산한 상태(eagerState)와 현재 상태(currentState)를 비교
      // 만약 두 상태가 객체이고 참조가 동일하다면 (또는 원시값이고 값이 동일하다면)
      if (Object.is(eagerState, currentState)) {
        // 상태가 변경되지 않았으므로, 실제 업데이트를 스케줄링하지 않고 종료 (bailout)
        // 이 경우, 불필요한 리렌더링을 방지할 수 있음
        recordEagerStateOnUpdate(update, eagerState); // 계산된 상태는 기록해둠
        return DID_NOT_SCHEDULE_UPDATE; // 업데이트 스케줄링 안 함을 알림
      }
    }
  }

  // 위의 최적화 조건에 해당하지 않거나, 상태가 변경된 경우:
  // 일반적인 업데이트 스케줄링 로직 수행
  root = enqueueConcurrentHookUpdate(fiber, queue, update, lane);
  if (root) {
    scheduleUpdateOnFiber(root, fiber, lane);
    return DID_SCHEDULE_UPDATE; // 업데이트 스케줄링 했음을 알림
  }
  return DID_NOT_SCHEDULE_UPDATE; // (예외적으로) 스케줄링 안됨
}
```

이 코드를 요약하자면 setState 호출했다고 무조건 리렌더링하는게 아니라 값이 실제로 바뀌었을 때만 리렌더링한다! 이말입니다.

결론적으로 state를 변경하고 싶으면 불변성을 지키면서 값을 변경해야 하는데, 객체나 배열은 스프레드 연산자를 사용해서 prevState의 값을 모두 복사한 새로운 참조를 생성하고, 그 위에 변경할 값을 덮어씌워야 한다.

### 1.3 상태 끌어올리기

상태 끌어올리기는 여러 컴포넌트가 동일한 상태를 공유하고 동기화해야할 때, 그 상태를 가장 가까운 공통 조상 컴포넌트로 옮겨 관리하는 리액트의 상태 관리 패턴입니다. 상태를 끌어올리면 하위 컴포넌트들은 상태를 직접 소유하는 대신 부모로부터 프롭스를 통해 데이터를 전달받습니다. 이로써 단일 출처 원칙이 지켜지며 애플리케이션 전체의 데이터 일관성을 유지할 수 있습니다.

```javascript

import { useState } from 'react';

// 아코디언 아이템 데이터 타입 정의
interface PoemItem {
  author: string;
  content: string;
}

interface AccordionItemProps extends PoemItem {
  isOpen: boolean; // ➊ 부모로부터 자신의 열림/닫힘 상태를 전달받음
  onClick: () => void; // ➋ 클릭되었을 때 호출할 함수를 부모로부터 전달받음
}

// AccordionItem: 상태를 소유하지 않음
const AccordionItem = ({
  author,
  content,
  isOpen,
  onClick,
}: AccordionItemProps) => {
  return (
    <div>
      <button
        onClick={onClick}
        style={{
          display: 'block',
          width: '100%',
          textAlign: 'left',
          padding: '10px',
          background: '#f1c40f',
          border: 'none',
          cursor: 'pointer',
        }}
      >
        {author}
      </button>
      {isOpen && (
        <div style={{ padding: '10px', border: '1px solid #f1c40f' }}>
          {content}
        </div>
      )}
    </div>
  );
};

interface AccordionProps {
  items: PoemItem[];
}
```

상태 끌어올리기를 사용했을 때 형제 컴포넌트 간의 공통된 상태가 있을 때 사용하면 정말 좋습니다.
그러나, 이벤트들을 매번 내려서 중첩이 깊어지는 경우들이 있습니다. 이 아코디언의 경우에서는 열림/닫힘의 자신 관심사를 처리 할 수 없어서, 떄에 따라서 제어 컴포넌트, 비제어 컴포넌트를 알맞게 사용하는게 중요합니다.

언제 어느 걸 선택해야 하나?

| 상황                                                    | 선택            |
| ------------------------------------------------------- | --------------- |
| 부모가 상태를 알아야 함 (여러 아이템 중 하나만 열림 등) | 제어 컴포넌트   |
| 컴포넌트가 완전히 독립적으로 동작해도 됨                | 비제어 컴포넌트 |
| 외부에서 열림/닫힘을 제어해야 함                        | 제어 컴포넌트   |
| 단순 UI 토글, 부모가 신경 안 써도 됨                    | 비제어 컴포넌트 |

## 11.3 컴포넌트 간의 데이터 흐름 돌아보기

방금 예시처럼 상위 레벨에서 하위 레벨로의 데이터 전달을 톱다운 방식의 데이터 흐름 혹은 단방향 데이터 흐름을 갖는다고 이야기하는데, 리액트는 기본적으로 단방향 흐름입니다.

### 1. 단방향 데이터 흐름과 단방향 바인딩

리액트는 부모 컴포넌트가 가진 데이터를 프롭스를 통해 내려주고, 자식 컴포넌트는 읽기 전용으로만 사용할 수 있습니다. 자식 컴포넌트에서 발생한 유저의 입력 등의 이벤트를 통해 변화를 알릴 수 있지만 데이터의 직접 변경은 부모에서 수행되어야합니다. 이러한 구조로 상태의 원인 추적과 동작 예측이 수월해집니다.

또한 상위 컴포넌트는 상태와 로직을 관리하고 하위 컴포넌트는 프롭스를 받아 표시 역할을 담당하므로 관심사가 분리됩니다. 자식으로 내려가는 프롭스 변화를 감지하여 불필요한 렌더링을 줄일 수 있습니다.

### 2. 양방향 데이터 흐름과 양방향 바인딩

Vue같은 프레임 워크에서는 양방향 데이터 바인딩을 지원하는데 모델과 뷰가 서로의 변경사항을 자동으로 반영합니다. 데이터 흐름이 양방향으로 일어나므로, 개발자가 별도의 이벤트 핸들러를 작성하지 않아도 UI와 상탯값이 항상 동기화됩니다.

```javascript
<script setup>
import { ref } from "vue"

const message = ref('hi');

</script>

<template>
  <input v-model="model" />
</template>
```

input 엘리먼트에 v-model을 사용하면 유저가 필드값을 입력할 때 message가 업데이트 되면서, message값을 바꾸면 input안에 있는 값도 업데이트 됩니다. 이러면 어디서 문제가 발생했는지 추적하기 조금 어려울 수도 있습니다.

## 11.4 리액트 배칭 돌아보기

버튼 클릭 한 번에 여러 상태가 연쇄적으로 변경되어야 하는 상황에서 리액트는 똑똑하게 모든 변경사항을 효율적으로 변경합니다.

### 1. 배칭의 개념과 필요성

배칭(batching)은 묶어서 한 번에 처리하는 기법인데, 리액트에서 배칭은 묶어서 한 번의 리렌더링만 수행하도록 합니다.

```javascript
import { FC, useState } from 'react';

// 무작위 헥스(HEX) 색상 코드를 생성하는 헬퍼 함수
const getRandomColor = () => {
  const letters = '0123456789ABCDEF'; // 헥스 코드에 사용될 문자들
  let color = '#'; // 헥스 코드 시작 문자
  for (let i = 0; i < 6; i++) { // 6자리 헥스 코드를 만들기 위해 반복함
    color += letters[Math.floor(Math.random() * 16)]; // 무작위로 문자 하나를 선택하여 색상 코드에 추가함
  }
  return color; // 완성된 헥스 색상 코드를 반환함
}

// DiceGame 컴포넌트의 props 타입을 정의함
interface DiceGameProps {
  maxTries?: number; // 최대 시도 횟수
}

// 주사위 게임 컴포넌트
const DiceGame: FC<DiceGameProps> = ({ maxTries = 5 } ) => {
    // 주사위 눈금 상태. 초기값은 1
  const [diceNumber, setDiceNumber] = useState(1);
    // 남은 시도 횟수 상태. 초기값은 props로 받은 maxTries 또는 기본값 5
  const [remainingTries, setRemainingTries] = useState(maxTries);
    // 배경색 상태. 초기값은 흰색('#FFFFFF')
  const [backgroundColor, setBackgroundColor] = useState('#FFFFFF');
    // 현재 테마 상태. 초기값은 'Default'
  const [theme, setTheme] = useState('Default');

    // 주사위를 굴리는 함수. 버튼 클릭 시 호출됨
  const rollDice = () => {
      // 1부터 6 사이의 무작위 주사위 눈금을 생성함
    const newDiceNumber = Math.floor(Math.random() * 6) + 1;
      // 남은 시도 횟수를 1 감소시킴
    const newRemainingTries = remainingTries - 1;
      // 새로운 무작위 배경색을 생성함
    const newBackgroundColor = getRandomColor();
      // 주사위 눈금에 따라 테마 이름을 결정함
      // 눈금이 6이면 'GoldenRabbit', 아니면 'random (시도 횟수)' 형태의 테마 이름을 가짐
    const newTheme = newDiceNumber === 6 ? 'GoldenRabbit' : `random ${maxTries - newRemainingTries}`;

    // 각 setState가 개별적으로 리렌더링을 트리거하는 것이 아니라,
    // 이벤트 핸들러 내의 모든 상태 업데이트가 끝난 후 단 한 번의 리렌더링만 발생시킴
    setDiceNumber(newDiceNumber); // 주사위 눈금 상태 업데이트
    setRemainingTries(newRemainingTries); // 남은 시도 횟수 상태 업데이트
    setBackgroundColor(newBackgroundColor); // 배경색 상태 업데이트
    setTheme(newTheme); // 테마 상태 업데이트

    // 이벤트 핸들러 내에서 상태 변경 직후의 값들을 콘솔에 기록함
    // 이 시점에서는 아직 리렌더링이 발생하기 전이므로, setXXX로 설정한 값들이
    // 실제 컴포넌트의 상태(diceNumber, remainingTries 등)에 반영되지 않았을 수 있음
    // 하지만 newXXX 변수를 통해 최신 값을 확인할 수 있음
    console.log('During event handler - Dice:', newDiceNumber, 'Tries:', newRemainingTries, 'Background:', newBackgroundColor, 'Theme:', newTheme);
  };
  // 컴포넌트 렌더링 시 현재 상태 값들을 콘솔에 기록함
  // 리액트 배칭으로 인해 rollDice 함수 내의 여러 setState 호출 후 이 로그는 한 번만 출력되어야 함
  console.log('Render - Dice:', diceNumber, 'Tries:', remainingTries, 'Background:', backgroundColor, 'Theme:', theme);

  // 주사위를 굴리는 함수 (비동기적 상태 업데이트)
  // setTimeout, Promise 등 비동기 콜백 내에서 상태 업데이트를 수행할 때도
  // 리액트 18부터는 자동 배칭이 적용되어 한 번의 리렌더링만 발생함
  const rollDiceAsync = () => {
    setTimeout(() => {
      // 1부터 6 사이의 무작위 주사위 눈금을 생성함
      const newDiceNumber = Math.floor(Math.random() * 6) + 1;
      // 남은 시도 횟수를 1 감소시킴
      // 주의: setTimeout 콜백이 실행될 시점의 remainingTries를 기준으로 계산하므로,
      // 여러 번 빠르게 클릭 시 의도치 않은 동작이 발생할 수 있음. (이는 배칭과는 다른 주제)
      // 함수형 업데이트 (setRemainingTries(prev => prev - 1))를 사용하면 이 문제를 해결할 수 있음.
      const newRemainingTries = remainingTries - 1;
      // 새로운 무작위 배경색을 생성함
      const newBackgroundColor = getRandomColor();
      // 주사위 눈금에 따라 테마 이름을 결정함
      const newTheme = newDiceNumber === 6 ? 'GoldenRabbit (Async)' : `random ${maxTries - newRemainingTries} (Async)`;

      // 비동기 콜백 내의 모든 상태 업데이트도 배칭되어 한 번의 리렌더링만 발생시킴 (리액트 18+)
      setDiceNumber(newDiceNumber);
      setRemainingTries(newRemainingTries);
      setBackgroundColor(newBackgroundColor);
      setTheme(newTheme);

      console.log('비동기 핸들러 내부 (setTimeout) - Dice:', newDiceNumber, 'Tries:', newRemainingTries, 'Background:', newBackgroundColor, 'Theme:', newTheme);
    }, 1000); // setTimeout으로 비동기 상황을 모방함
  };
  return (
      // 배경색은 backgroundColor 상태 값에 따라 동적으로 변경됨
    <div style={{ backgroundColor, padding: '20px', textAlign: 'center' }}>
      <h1>Dice Game</h1>
        {/* 현재 주사위 눈금을 표시함 */}
      <p>Current Dice Number: {diceNumber}</p>
        {/* 남은 시도 횟수를 표시함 */}
      <p>Remaining Tries: {remainingTries}</p>
        {/* 현재 테마를 표시함 */}
      <p>Current Theme: {theme}</p>
        {/* 버튼 클릭 시 rollDice 함수를 호출함. 남은 시도 횟수가 0이면 비활성화됨 */}
      <button onClick={rollDice} disabled={remainingTries === 0}>주사위 던지기 게임</button>
        {/* 버튼 클릭 시 rollDiceAsync 함수를 호출함. 남은 시도 횟수가 0이면 비활성화됨 */}
      <button onClick={rollDiceAsync} disabled={remainingTries === 0}>주사위 던지기 게임(비동기)</button>
    </div>
  );
}

export default DiceGame; // DiceGame 컴포넌트를 기본으로 내보냄
```

리액트는 즉 setState()가 호출될 떄 마다 즉시 리렌더링을 발생시키는 것이 아니라, 내부의 업데이트 큐에 요청을 쌓아두었다가 주로 이벤트 핸들러 실행이 끝난 이후 한 번에 처리하여 단일 리렌더링을 수행합니다.

### 2. 비동기 동작에서의 배칭 프로세스

리액트 17까지는 `setTimeout`, `Promise`, 네이티브 이벤트 핸들러 안에서는 배칭이 적용되지 않았습니다. 그래서 비동기 콜백 안에 setState가 3개 있으면 리렌더링이 3번 발생했습니다.

리액트 18부터는 자동 배칭(Automatic Batching) 이 도입되어, 비동기 콜백 안에서도 모든 setState를 모아 한 번의 리렌더링만 발생시킵니다. 위 `rollDiceAsync`의 `setTimeout` 내부에서 4개의 setState를 호출해도 리렌더링은 단 한 번만 일어납니다.

### 3. react-dom의 flushSync()

배칭이 항상 좋은 건 아닙니다. 간혹 "이 setState는 지금 당장 즉시 리렌더링을 유발해야 한다"는 상황이 생깁니다. 이때 `flushSync()`를 사용하면 배칭을 강제로 중단하고 즉시 DOM을 업데이트할 수 있습니다.

```javascript
import { flushSync } from "react-dom";

flushSync(() => {
  setCount(1); // 이 블록이 끝나는 즉시 리렌더링 발생
});
// 이 시점에서 DOM은 이미 업데이트된 상태
flushSync(() => {
  setFlag(true); // 또 한 번의 리렌더링 발생
});
```

대표적인 사용 사례는 DOM 측정이 필요한 경우입니다. 상태를 바꾼 직후 `scrollHeight` 같은 DOM 값을 읽어야 할 때, 배칭으로 인해 DOM이 아직 업데이트되지 않은 상태에서 읽으면 이전 값이 나옵니다. `flushSync`로 강제로 먼저 반영시킨 뒤 DOM을 읽으면 정확한 값을 얻을 수 있습니다.

단, `flushSync`는 성능에 영향을 주므로 꼭 필요한 경우에만 사용하는 것이 좋습니다.
