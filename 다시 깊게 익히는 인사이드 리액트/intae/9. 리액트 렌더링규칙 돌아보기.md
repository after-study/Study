# 9장. 리액트 렌더링 규칙

## 9.1 리액트 렌더링 규칙을 돌아봐야 하는 이유

리액트 개발자라면 누구나 렌더링과 관련된 예상치 못한 문제를 경험하십니다. 컴포넌트 상단에 console.log를 넣었을 뿐인데 버튼을 클릭할 때마다 로그가 찍히거나, props가 변경되지 않았는데 자식 컴포넌트가 불필요하게 리렌더링되는 상황이 대표적입니다.
이런 문제의 근원은 렌더링 메커니즘에 대한 미묘한 오해에서 비롯됩니다. 많은 개발자가 리렌더링을 프롭스나 상태가 변경될 때만 발생한다고 생각하지만, 이는 절반만 맞는 이야기입니다. 렌더링 규칙을 제대로 이해하면 성능 최적화 기술과 프롭스 불변성을 왜 지켜야 하는지까지 자연스럽게 이해하실 수 있습니다.

## 9.2 렌더링 조건

리액트 컴포넌트의 렌더링을 유발하는 조건은 네 가지로 정리됩니다.

- 최초 렌더링 — 애플리케이션이 처음 로드될 때
- 상태 변경 — useState()의 상태 변경 함수나 useReducer()의 dispatch 호출 시
- 부모 컴포넌트의 리렌더링 — 부모가 렌더링되면 자식도 기본적으로 함께 렌더링됩니다
- 컨텍스트 변경 — useContext()로 구독 중인 컨텍스트값이 변경될 때

### 조정자(Reconciler)와 렌더러(Renderer)

- 조정자: 가상 DOM과 재조정 알고리즘을 담당하는 리액트의 뇌입니다. 어떤 부분이 변경되어야 하는지 계산합니다.
- 렌더러: 조정자로부터 변경 사항을 전달받아 실제 UI로 그려내는 손과 발입니다. react-dom(웹), react-native(모바일) 등이 있습니다.

### 애플리케이션이 처음 로드될 때

모든 렌더링은 최초 렌더링에서 시작됩니다. ReactDOM.createRoot()로 루트 DOM 노드를 지정하고 render()를 호출하면 리액트 애플리케이션이 실제 DOM에 연결됩니다.

### 컴포넌트 내부 상태값의 변경으로 인한 리렌더링

컴포넌트는 각자 고유의 상태를 가지며, 이 값이 변경될 때 UI를 업데이트하기 위해 리렌더링이 발생합니다.

- useState() — setState() 호출 시 상태 업데이트가 예약되고 리렌더링이 발생합니다
- useReducer() — dispatch() 함수를 통해 reducer 내부의 조건에 따라 상태가 업데이트됩니다
- 함수형 컴포넌트의 forceUpdate — 공식 API가 없으므로 useState()를 활용하여 구현합니다

### 부모 컴포넌트가 리렌더링되었을 때

꼭 알아야 할 핵심: 부모 컴포넌트가 리렌더링되면, 자식에게 전달되는 프롭스가 변경되었는지 여부와 관계없이 자식 컴포넌트들도 모두 리렌더링됩니다. 이것이 기본 동작입니다.

```jsx
// 부모 리렌더링 시 자식의 기본 리렌더링 동작
const Child = ({ value }) => {
  console.log(`Child 컴포넌트 렌더링 (value: ${value})`);
  return <div>Child (props: {value})</div>;
};

const Parent = () => {
  const [count, setCount] = useState(0);
  // count가 변경되면 Parent가 리렌더링되고 Child도 함께 리렌더링됨
  // ...
};
```

### 컨텍스트의 값이 변경되었을 때

useContext()를 사용해 컨텍스트를 구독하는 컴포넌트는 Provider의 value가 변경될 때마다 리렌더링됩니다.

주의할 점: 컴포넌트가 value 객체 안의 특정 값(mood)만 사용하든, 특정 함수(changeMood())만 사용하든 관계없이 리렌더링이 일어납니다. 따라서 컨텍스트를 설계할 때 불필요한 리렌더링이 발생하지 않도록 구조를 잘 설계해야 합니다.

```jsx
// DisplayMood는 mood만, ToggleMoodButton은 changeMood만 사용하지만
// Provider의 value가 변경되면 둘 다 리렌더링됨
const DisplayMood = () => {
  const { mood } = useContext(MoodContext);
  // ...
};

const ToggleMoodButton = () => {
  const { changeMood } = useContext(MoodContext);
  // ...
};
```

## 9.3 리렌더링에 대한 오해

리액트의 렌더링 기본 원칙: 부모 컴포넌트가 리렌더링되면, 그 자식 컴포넌트도 리렌더링됩니다. 하지만 이 원칙이 프롭스나 상태 변경에 가려져 개발자들이 몇 가지 중요한 오해에 빠지기 쉽습니다.
9.3.1 자식 컴포넌트와 props.children

핵심 차이점: props.children으로 전달된 요소는 부모가 리렌더링되어도 반드시 함께 리렌더링되지는 않습니다.

```jsx
const App = () => (
<Parent>
<RerenderChecker /> {/_ children으로 전달 _/}
</Parent>
);

<ChildComponent />
//  — Parent의 return문 내부에 있으므로 Parent가 리렌더링될 때마다 매번 새로운 React 엘리먼트로 생성 → 항상 리렌더링

<RerenderChecker />
// (children) — 상위 App 스코프에서 생성되므로 Parent가 리렌더링되어도 App이 리렌더링되지 않는 한 참조가 동일하게 유지 → 리렌더링 안 됨
```

이것은 props.children을 활용하여 렌더링 스코프를 분리하고 불필요한 리렌더링을 방지하는 강력한 최적화 패턴입니다.

9.3.2 컴포넌트에 전달되는 프롭스값이 변경되었을 때?
프롭스의 변경 자체는 컴포넌트의 리렌더링을 트리거하는 요소로 작용하지 않습니다. 변경된 프롭스를 컴포넌트에 적용하려면 결국 부모 컴포넌트가 리렌더링되어야 합니다.
useRef()를 활용한 극단적 예시: ref 값을 1초마다 변경하면서 이를 자식의 prop으로 전달해도, Parent가 리렌더링되지 않으면 Child는 새로운 props를 전달받지 못해 리렌더링이 일어나지 않습니다.

```jsx
const Parent = () => {
  const propRef = useRef({ value: 0 });

  useEffect(() => {
    const interval = setInterval(() => {
      propRef.current.value += 1; // ref 변경 → 리렌더링 유발 안 됨
    }, 1000);
    return () => clearInterval(interval);
  }, []);

  // propRef.current.value가 메모리에서 바뀌어도
  // Parent가 리렌더링되지 않으므로 Child에게 새 props가 전달되지 않음
  return <Child value={propRef.current.value} />;
};
```

### 9.4 리액트 렌더링 규칙

리액트 컴포넌트가 동일한 데이터를 받았음에도 실행할 때마다 다른 모습을 보여준다면, 개발자는 혼란에 빠지고 버그 추적이 매우 어려워집니다. 리액트는 이런 예측 불가능성을 방지하기 위해 렌더링 규칙을 적용합니다.

### 부수 효과 제거와 멱등성 보장

렌더링은 순수 함수여야 합니다. 같은 인풋에 대해 항상 같은 결과를 반환하고, 외부 자원을 사용하지 않아야 합니다.

순수 함수와 멱등성:

순수 함수: 같은 인수에 대해 항상 같은 결과를 반환하고 외부 변수를 변경하지 않습니다
멱등성: 같은 인풋에 의해 여러 번 실행되어도 항상 같은 아웃풋을 보장합니다

```jsx
// ✅ 순수 함수 (멱등성 보장)
function getSquareRoot(x) {
  return Math.sqrt(x);
}

// ❌ 순수 함수가 아님 (멱등성 미보장)
function getRandomNumber() {
  return Math.random();
}
```

렌더링 중 피해야 할 부수 효과들:

부수 효과

- 콘솔 로깅외부 시스템 이용 (단, 디버깅 목적의 console.log()는 예외적으로 허용)쿠키/저장소 접근외부 환경에 의존하여 순수성이 깨집니다
- 상태 업데이트 (setState)렌더링 중 호출 시 무한 렌더링 루프에 빠질 위험이 있습니다
- 네트워크 요청비동기 작업으로 렌더링 시점을 예측할 수 없게 만듭니다
- 외부 변수 변경다른 컴포넌트의 동작에 영향을 줄 수 있습니다
- 직접적인 DOM 조작React의 가상 DOM 렌더링 메커니즘과 충돌합니다

```jsx
// ❌ 렌더링 중 부수 효과가 있는 나쁜 예
const SideEffectComponent = () => {
const [state, setState] = useState("");

console.log("Rendering…"); // 콘솔 로깅
const token = document.cookie.split('; ')... // 쿠키 접근
if (token) { setState(token); } // 렌더링 중 상태 업데이트 (위험!)
axios.get("/data"); // 네트워크 요청
if (token) { items.push(token); } // 외부 변수 변경
document.getElementById("main")... // 직접 DOM 조작

// ✅ 실제 애플리케이션에서는 모든 부수 효과는 useEffect 훅 내부에서 처리해야 합니다
return <div>Do not use side effects during rendering</div>;
};
```

### 리액트 렌더링 규칙 - JSX로 전달된 값의 불변성을 유지

꼭 알아야 할 핵심: JSX에 전달된 객체나 배열은 절대 직접 수정해서는 안 됩니다. 변경이 필요하다면 항상 원본을 유지한 채 새로운 객체나 배열을 생성하여 상태 변화를 표현해야 합니다.

```jsx
// ❌ 나쁜 예: 전달된 객체를 직접 수정
function GoldenRabbitPageBad({ color }) {
  const styles = { color, sparkles: "✨", size: "large" };
  const rabbitHeader = <RabbitHeader styles={styles} />;

  // ✖ 이미 JSX에 전달된 객체를 직접 수정 → 불변성 원칙 위반
  styles.size = "small";

  return <RabbitContent header={rabbitHeader} styles={styles} />;
}

// ✅ 좋은 예: 스프레드 연산자로 새로운 객체 생성
function GoldenRabbitPageGood({ color }) {
  const headerStyles = { color, sparkles: "✨", size: "large" };
  const rabbitHeader = <RabbitHeader styles={headerStyles} />;

  // ✔ 스프레드 연산자로 새 객체를 생성하여 원본 불변성 보장
  const contentStyles = {
    ...headerStyles,
    size: "small",
  };

  return <RabbitContent header={rabbitHeader} styles={contentStyles} />;
}
```

불변성을 유지하는 것은 애플리케이션의 안정성과 성능을 보장하는 필수적인 원칙이자 좋은 습관입니다. 특히 복잡한 애플리케이션에서는 불변성 원칙 준수가 디버깅을 용이하게 하고 예측 가능한 동작을 보장하는 데 결정적인 역할을 합니다.

## 함수형 컴포넌트 렌더링/커밋 단계 도식화

```

┌─────────────────────────────────────────────────────────┐
│ ① 렌더 단계 │ ② 커밋 단계 │
│ │ │
│ 마운트하는 중 │ 리액트가 DOM을 │
│ function() {} │ 업데이트합니다. │
│ ├─ useMemo() │ │
│ └─ return () │ 언마운트 중 │
│ │ 클린업 함수 실행 │
│ 업데이트 중 │ │
│ useState() useReducer() │ useEffect() │
│ useCallback() useContext() │ useLayoutEffect() │
└─────────────────────────────────────────────────────────┘

핵심 정리

리렌더링은 상태/프롭스 변경 때만 일어나는 게 아닙니다.
부모가 리렌더링되면 자식도 기본적으로 리렌더링됩니다
props.children 패턴으로 렌더링 스코프를 분리하면 불필요한 리렌더링을 방지할 수 있습니다
프롭스 변경 자체가 리렌더링의 트리거가 아니라, 부모의 리렌더링이 근본 원인입니다
렌더링 함수는 순수 함수여야 하며, 부수 효과는 반드시 useEffect 내부에서 처리합니다
JSX에 전달된 객체/배열의 불변성을 반드시 유지하고, 변경이 필요하면 새로운 객체를 생성합니다
Context를 구독하는 컴포넌트는 value 객체의 어떤 속성을 사용하든 Provider의 value가 변경되면 모두 리렌더링됩니다
```
