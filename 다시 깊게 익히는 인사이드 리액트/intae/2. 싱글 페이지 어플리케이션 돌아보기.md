기존 MPA 방식으로 만들었었다면 더 나은 사용자 경험을 제공하기 위해서 SPA 방식이 나왔다.

리렌더링이 필요할 때 (데이터가 업데이트 되어야할 때) 페이지를 서버에서 전체 요청하는 것이 아닌, 클라이언트에서 변경된 컴포넌트 단위의 부분만 리렌더링하여 유저 경험을 향상시키고 생산성을 높이고 있습니다.

SPA의 장점을 쭉 알아봅시다.

### 1. 싱글 페이지 어플리케이션의 장점

SPA의 등장은 프론트엔드와 백엔드의 역할을 명확하게 분리시키는 계기가 되었습니다.

MPA는 a태그의 링크를 클릭해서 이동하였을 때 링크에 해당하는 페이지를 요청하고, 렌더링됩니다. 그 과정에서 html, css, js까지 전부 리로드 되죠, 하지만 SPA는 대부분 재사용하기 때문에 리로드 시간이 줄고, 서버와 클라이언트간의 네트워크 요청도 줄입니다. 또한 페이지 전체가 바뀌는 것이 아니기 때문에 부드러운 사용자 경험도 제공합니다.

### 2. 싱글 페이지 어플리케이션 단점 극복하기

MPA에 필요한 CSS,JS,HTML등을 하나의 JS에 몰아 넣으니 느려지지 않을까? 라는 생각도 드었지만 Lazy Loading 이라는 개념을 적용하여 페이지 전체에 모든 Js 를 로드하는 것이 아니라 특정 코드 블록만 필요할 떄 로드할 수 있도록 만들었고, 이렇게 한 번 받아온 파일은 캐싱될 수 있게 만들었습니다.
