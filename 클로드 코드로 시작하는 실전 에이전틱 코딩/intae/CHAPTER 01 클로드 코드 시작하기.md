# CHAPTER 01 클로드 코드 시작하기

## 1-1 클로드 코드와 에이전틱 코딩

### 에이전틱 AI와 MCP

- **에이전틱 AI(agentic AI)**: 질문에 답하는 수준을 넘어, 스스로 문제를 분석하고 도구를 활용해 자율적으로 작업을 수행하는 AI 시스템. 기존 AI가 "무엇을 할까요?"라면, 에이전틱 AI는 "이렇게 하겠습니다"라고 제안하고 실행까지 완수한다.
- **MCP(Model Context Protocol)**: AI 모델이 외부 데이터 소스 및 도구와 상호작용하도록 하는 표준 프로토콜. 다양한 장치를 컴퓨터에 연결하는 USB 표준에 비유된다. 구글 드라이브 문서 읽기, 지라 티켓 업데이트 등이 MCP 서버 연동으로 구현된다.
- 클로드 코드는 앤트로픽의 에이전틱 전략의 산물로, 개발자 워크플로에 직접 통합되는 **에이전틱 코딩 CLI 도구**다.

### 에이전틱 코딩이란

- **에이전틱 코딩(agentic coding)**: AI가 파일 시스템 탐색, 코드 수정, 테스트 실행까지 자율로 수행하는 접근 방식. 개발자가 높은 수준의 목표를 제시하면 AI가 이를 하위 작업으로 분해하고 실행 계획을 수립해 완수한다.
- 앤트로픽은 이 과정을 **탐색-계획-구현-커밋(Explore-Plan-Implement-Commit) 워크플로**로 정의한다.

#### 에이전틱 코딩 워크플로와 클로드 코드 도구

```
사용자 요청 → Explore(코드 탐색) → Plan(계획 수립) → Code(구현) → Test(검증) → Commit(커밋)
```

대응 도구: 파일 편집 / 명령 실행 / 깃 통합 / MCP 연동

- **커서(Cursor)**: VS Code 기반 에디터에 에이전트 모드를 통합한 IDE형 도구.
- **클로드 코드**: 에디터 없이 터미널에서 작동하는 CLI 기반 도구. 파일 읽기·쓰기, 명령 실행, 깃 커밋 생성을 내장 도구(Read, Write, Bash)로 직접 수행한다. MCP 서버로 깃허브·피그마·슬랙 등 외부 서비스까지 확장 가능. CI/CD 파이프라인이나 자동화 스크립트에 직접 편입할 수 있다는 점이 구조적 차별점이다.

### 에이전틱 루프: 클로드 코드의 동작 원리

순환 구조 3단계로 작동한다(고정된 순서가 아니라 상황에 따라 유동적으로 결합).

1. **컨텍스트 수집**: 프롬프트를 받으면 즉시 코딩하지 않고, 프로젝트 파일 구조 탐색·관련 코드 검색(Grep, Glob)·`CLAUDE.md` 규칙 로딩으로 필요한 정보를 수집한다.
2. **작업 수행**: 실행 계획을 세우고 도구를 호출한다. 파일 조작(Read, Write, Edit, MultiEdit), 검색(Glob, Grep), 셸 실행(Bash), 웹 접근(WebFetch, WebSearch), 서브에이전트 생성, 작업 목록 관리(TodoWrite) 등. `/model` 명령으로 Opus·Sonnet 선택 시 High/Medium/Low 추론 깊이를 선택해 추론 예산(thinking budget)을 조절한다.
3. **결과 확인**: 자신의 작업을 스스로 검증한다(테스트 실행, 린트, 빌드). 검증 방식은 규칙 기반/시각적/LLM 기반 피드백으로 분류된다. 문제 발견 시 1단계로 돌아가 수정을 반복한다.

> 이 반복 구조는 TDD 워크플로에서 명확히 드러난다. 먼저 테스트를 작성하고, 실패하면 통과하도록 코드를 구현하며, 통과할 때까지 반복한다.

- **서브에이전트**: 메인 에이전트가 Agent 도구로 서브에이전트를 생성하면, 독립된 컨텍스트 윈도우에서 탐색을 수행하고 결과를 요약(1,000~2,000 토큰 수준)해 반환한다. 수만 토큰 규모의 탐색도 메인 컨텍스트를 효율적으로 관리할 수 있다.

### 터미널에서 시작하는 AI 코딩

- CLI 인터페이스가 차별점이며, **모듈성(modularity)**이라는 유닉스 철학을 따른다. 파이프(`|`)로 다른 유닉스 도구와 결합해 워크플로를 구성한다.

```bash
$ cat security_logs_*.log | claude -p "이 로그들을 분석해서 해킹 시도 패턴을 찾아서 위협 수준과 함께 보고해 줘."
```

- 세 가지 요소: ① `cat ... *.log`(와일드카드로 여러 로그 일괄 처리) ② `|`(파이프, 앞 출력을 다음 입력으로 전달) ③ `claude -p`(비대화형 모드로 프롬프트 실행). GUI 도구나 플러그인 설치 없이 자동화 솔루션을 만드는 것이 클로드 코드의 철학이다.

### 컨텍스트 윈도우와 20만 토큰 컨텍스트의 의미

- **컨텍스트 윈도우(context window)**: 한 번의 추론에서 참조할 수 있는 토큰의 총량. 적재 순서는 `CLAUDE.md` 기반 메모리(엔터프라이즈 정책 → `./CLAUDE.md` 프로젝트 메모리 → `~/.claude/CLAUDE.md` 사용자 메모리 → 프로젝트별 로컬 메모리) → 대화가 진행되며 프롬프트·응답·도구 호출 결과가 누적.
- 클로드 코드는 **20만 토큰** 컨텍스트 윈도우 지원(Opus 4.6은 100만 토큰 베타 지원, 20만 초과 시 별도 API 요금).
- 20만 토큰을 모두 채우는 것이 좋은 것은 아니다. 앤트로픽은 〈AI 에이전트를 위한 효과적인 컨텍스트 엔지니어링〉에서 "원하는 결과의 가능성을 최대화하는 최소한의 고신호 토큰 집합을 찾는 것"을 핵심 원칙으로 제시한다. 토큰 수가 늘면 정보 회상 정확도가 떨어지고 장거리 추론 능력이 약화된다.
- 트랜스포머에서 모델은 토큰 간 관계 계산에 **주의력 예산(attention budget)**을 갖는데, 새 토큰이 추가될 때마다 분산된다. 500줄 파일 10개를 모두 읽으면 5만~10만 토큰이지만, 필요한 부분만 50줄씩 읽으면 5천~1만 토큰으로 동일한 작업이 가능하다.

> **Tip** 컨텍스트는 한정된 자원이다. 상황에 맞는 관리 전략으로 토큰 소비를 최적화해야 한다.

### 컨텍스트 윈도우 관리 전략 (5가지)

1. **컴팩션(compaction)**: 사용량이 한계의 98%에 도달하면 이전 대화를 자동 요약한다. 아키텍처 결정·미해결 버그·핵심 구현 세부는 보존하고 중복·오래된 도구 출력은 제거한다.
2. **`/clear` 명령**: 대화 기록을 완전히 초기화한다. 서로 관련 없는 작업을 연속 수행할 때 유용(예: 인증 모듈 수정 후 프런트엔드 UI 작업으로 전환).
3. **점진적 공개(progressive disclosure)**: 가벼운 식별자(파일 경로, 링크)만 유지하다 필요할 때 동적 로드한다. 스킬(Skills) 시스템이 이 원칙을 따른다(초기 ~100 토큰 메타데이터만 로드 → 트리거 조건 충족 시 본문 로드).
4. **서브에이전트 활용**: 각 서브에이전트는 별도의 20만 토큰 컨텍스트를 사용한다. 백엔드·프런트엔드를 각각 독립 탐색 후 메인 에이전트가 두 요약을 종합한다.
5. **구조화된 메모 작성**: 컨텍스트 윈도우 외부에 마크다운 파일로 체크리스트를 작성하고 다시 불러온다. 대규모 마이그레이션·다단계 수정에서 진행 상황 추적에 적합하다.

### 다섯 가지 접근 영역과 핵심 기능

- 클로드 코드는 개발자의 실제 작업 환경에 직접 접근한다. 다섯 가지 접근 영역이 하나의 에이전틱 루프 안에서 유기적으로 연결된다.
  - **MCP 서버 접근**(예: 센트리에서 에러 데이터 수집)
  - **파일 시스템 접근**(관련 소스 코드 읽기)
  - **웹 접근**(라이브러리 최신 문서 확인)
  - **터미널 접근**(코드 수정 후 테스트 실행)
  - **에디터 컨텍스트**(IDE 진단 결과 확인)
- **보안 제어**: `settings.json`의 `permissions` 설정으로 특정 도구 허용/차단, 샌드박스 모드로 파일 시스템·네트워크 접근 격리(자세한 내용은 2장).

## 1-3 클로드 코드 CLI 기초

- CLI 명령어는 파이프(`|`)로 다른 명령과 결합해 강력한 자동화 워크플로를 형성한다.
- 명령어 구조는 기본 명령어, 플래그, 슬래시 명령어, 환경 변수가 서로 다른 수준에서 동작을 제어한다.

### 클로드 코드 CLI 명령 체계

```
claude 명령
   └── 실행 모드
        ├── 플래그 없음 → 대화형 모드 → /help, /clear 등 슬래시 명령
        ├── -p          → 비대화형 모드 → 결과 출력 후 종료
        └── -c          → 세션 재개 → 이전 대화 복원
```

### 대화형 모드와 비대화형 모드

- **대화형 모드**: 플래그 없이 `claude` 명령만으로 실행하는 REPL(Read-Eval-Print Loop) 스타일 세션. 탐색 작업과 장기 세션에 적합. 실행 디렉터리를 작업 디렉터리로 인지하므로 항상 작업 디렉터리로 이동 후 실행을 권장한다.

```bash
$ cd /path/to/your/project
$ claude
```

- 세션 시작 시 프로젝트 구조를 자동 인식하고, 깃 저장소면 히스토리를 참조하며, 루트의 `CLAUDE.md`를 자동으로 읽어 컨텍스트로 활용한다.
- **비대화형 모드(헤드리스 모드)**: `-p` 또는 `--print` 플래그로 실행. 프롬프트 결과만 출력 후 종료. 스크립팅·자동화에 적합.

```bash
$ claude -p "이 함수의 복잡도를 분석해 줘."
```

### 설정 디렉터리 구조

```
project-root/
├── .claude/
│   ├── commands/        # 커스텀 슬래시 명령어
│   ├── agents/          # 커스텀 에이전트 정의
│   ├── skills/          # 에이전트 스킬 모듈
│   ├── hooks/           # 이벤트 훅 스크립트
│   ├── output-styles/   # 출력 스타일 템플릿
│   └── settings.json    # 프로젝트별 클로드 코드 설정
└── CLAUDE.md            # 프로젝트 지침서(컨텍스트로 자동 로드)
```

- 세션 종료: `exit` 또는 `Ctrl + C`. 세션 상태는 자동 저장되어 재개 가능.

### 필수 슬래시 명령어

- `/help`: 사용 가능한 모든 명령어와 단축키 확인.
- `/clear`: 현재 대화 컨텍스트 초기화. 세션 내 대화 기록만 지우며 프로젝트 설정·체크포인트는 유지. 필요 시 `/rewind`로 이전 상태 복구.
- `/exit`: 세션 종료(`Ctrl + C`도 가능). 종료 시 현재 상태가 자동 저장되어 `claude -c`로 재개 가능.

### 다양한 실행 플래그

- **일회성 작업 실행**: `claude "질문"` — 별도 플래그 없이 즉시 실행.
- **비대화형 쿼리 실행**: `claude -p "질문"` (`-p`, `--print`) — 결과만 출력 후 종료.
- **최근 대화 세션 재개**: `claude -c` (`-c`, `--continue`) — 이전 작업 맥락을 그대로 유지해 마지막 상태로 복귀.
- **이전 대화 세션 중 선택 재개**: `claude -r` (`-r`, `--resume`) — 여러 세션 목록 중 특정 시점 선택.

> **Tip** 세션을 시작할 때는 프로젝트 디렉터리에서 `claude` 명령을 실행하자.

### 시스템 상태 진단과 문제 해결

- `claude doctor` 명령으로 설치 유형, 경로, 설정 상태(설치 방식·버전·검색 엔진·자동 업데이트·버전 락 등)를 진단한다. API 키 누락, 네트워크 연결 실패, 권한 등 문제 원인을 빠르게 파악할 수 있다.

### 체크포인트와 되감기 시스템

- **체크포인트(checkpointing)**: 파일 편집을 자동 추적하고 원치 않는 변경을 이전 상태로 되돌리는 세션 수준 안전망. 프롬프트를 입력할 때마다 새 체크포인트가 자동 생성된다.
- 세션 전체에 걸쳐 지속되며 `-c`/`-r`로 이전 세션 체크포인트에도 접근 가능. 세션 저장 기간은 30일(`settings.json`의 `cleanupPeriodDays`로 조정).
- `Esc` 두 번 또는 `/rewind` 명령으로 되감기 메뉴를 연다.

#### 체크포인트의 복구 옵션 (3가지)

- **대화만**: 코드 변경은 유지하고 사용자 메시지로 되감기.
- **코드만**: 대화는 유지하고 파일 변경만 되돌리기.
- **코드와 대화 모두**: 둘 다 복구해 특정 시점부터 다시 시작.

#### 체크포인트가 유용한 상황

- 시작점을 잃지 않고 대체 구현을 탐색해야 할 때.
- 잘못된 변경으로 버그가 생기거나 기능이 깨져 빠르게 복구해야 할 때.
- 작동하는 현 상태를 유지하면서 실험해야 할 때.

#### 체크포인트의 한계

1. **Bash를 통한 파일 변경은 추적하지 못한다**: Edit/Write 도구 수정은 기록되지만 `rm`, `mv`, `cp`, `sed` 등 셸 명령 변경은 기록되지 않는다. 파일 수정은 되도록 전용 편집 도구를 사용하도록 유도하는 편이 안전하다.
2. **외부 변경 사항은 기록되지 않는다**: 별도 편집기·다른 터미널·다른 클로드 코드 인스턴스의 변경은 미기록(병렬 실행 고급 워크플로에서 주의).
3. **버전 관리 시스템의 대체재가 아니다**: 영구 이력·브랜치·협업에는 반드시 깃을 사용. 체크포인트는 커밋과 커밋 사이의 보완 도구다.

## 1-4 프로젝트로 특징 이해하기

> 명령어 앞의 프롬프트 기호로 실행 환경을 구별: `$`는 클로드 코드 진입 전 셸에서 실행하는 명령, `>`는 클로드 코드 진입 후 실행하는 명령.

### 프로젝트 컨텍스트의 이해

- 가장 먼저 이해해야 할 핵심 개념은 **컨텍스트 인식**이다. 클로드 코드는 `claude`가 실행되는 디렉터리를 기준으로 프로젝트 전체 구조(파일 목록·깃 히스토리·설정 파일)를 자동 스캔한다. 반드시 프로젝트 디렉터리로 이동 후 실행해야 한다.
- 컨텍스트 인식 확인용 질문 예시:

```
> 이 프로젝트에 대해서 분석해 줘.
> 이 프로젝트의 기술 스택은 어떻게 구성이 되어 있는지 알려줘.
> 이 프로젝트의 전체 디렉터리/파일 구조에 대해서 상세히 분석해서 보고서 형식으로 마크다운 문서를 생성해 줘.
```

### 간단한 프로그램 작성

- 빈 디렉터리에서 시작해 **플랜 모드(plan mode)**로 프로젝트 초기 구조를 생성하도록 요청할 수 있다. 플랜 모드에서는 "CLI 인터페이스가 필요한가요?", "테스트 코드도 함께 작성할까요?" 같은 질문으로 요구 사항을 명확히 한 후 최종 계획을 수립한다.

```bash
$ claude                       # ① macOS 터미널 / 윈도우 명령 프롬프트에서 claude 입력
> 1. Yes, I trust this folder  # ② 디렉터리를 신뢰한다고 선택
> /plan                        # ③ 플랜 모드로 변경
> 테스트 코드 넣어서 구구단 프로그램을 작성해 줘.   # ④ 프로그램 작성 요청 프롬프트 입력
```

- 이후 클로드 코드가 계획하고 에이전트를 생성해 구현 후 파일 목록과 실행 방법을 안내한다. 테스트 코드 생성과 예외 처리까지 포함한 완성도 높은 프로그램을 한 번의 대화로 구현한다.
- TDD를 따르려면 먼저 테스트를 작성하고 실패를 확인한 뒤 코드를 구현하도록 요청한다. 플랜 모드로 워크플로를 체계적으로 관리해 일관된 품질을 유지할 수 있다.

### 파일 읽기와 쓰기 패턴

- 클로드 코드는 내부적으로 Read, Write, Edit, Bash 등의 전용 도구를 호출해 파일 시스템에 접근한다.

#### Read: 지능적인 파일 읽기

- 컨텍스트 효율을 위해 파일을 읽을 때 **필요한 부분만 선택적으로 읽는다**. 대용량 파일은 전체를 읽지 않고 요청 관련 특정 섹션만 추출해 토큰 소모를 최적화한다.
- `@` 기호로 특정 파일을 명시적으로 참조하도록 지시할 수 있다.

```
> @config.yaml의 설정을 살펴보고 개선점을 제안해 줘.
```
